## Gotchas & decisions
- turbo-freq disable / core-power disable make everything read "disabled" but never clear CLOS configs or per-core assocs — readback of enable bits is misleading; check get-config/get-assoc instead.
- 3af6's earlier "success" was leftover experimental state (June core-power runs rewrote its CLOS table in one long boot), not a hardware/BIOS difference; ILOM fault SPINTEL-8008-5L, TPM, C-states, microcode, power caps were all ruled out.
- Boot-default CLOS0 membership = 16 PCT cores plus HT siblings (32 CPUs); everything else is CLOS3 (max 2700).
- One assoc call covers all CPUs (--cpu 0-287); no per-package loop needed (user-corrected).
- intel-speed-select writes results to stderr and reports success per entity — scripts must capture 2>&1 and count exact success lines (288 for the full assoc) plus check for fail/error markers, or partial failures pass silently.
- sudo cannot invoke sourced shell functions; run flat_freq_apply etc. without sudo — they call sudo -n <isst binary> internally (hence the per-host sudoers requirement).
- Reverting requires restoring the exact boot assoc map (CLOS0 ranges hard-coded in flat_freq_utils.sh from verified sweeps), then re-enabling turbo-freq; both machines' reverts were verified this session.
- Scratchpad copies of the harness are ephemeral; canonical copies live in the NFS debug_3bda directory (as of session end).
