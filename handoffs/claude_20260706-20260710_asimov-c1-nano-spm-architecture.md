## Objective

Understand, from the C1-Nano software perspective, how software on the ARM
cores accesses the SPM's 16 MiB data memory and 4 MiB metadata (RG data):
data formats, data paths, what is transparent to software — then pin down
the exact routing-data structure, and extend the same treatment to the
SAL/SPA and SVL blocks. Deliverables are one-page HTML notes (one per
block), a markdown data-structure model following the user's Notion "SPM
model" template, and an SVG register diagram.

  metadata records through the CMN AXI ports); (3) registers go via CMN
  SBSX -> AXI4-Lite 64-bit -> spm_axil_regs (host window 4 KB at
  0x234000). Metadata = "RG data" = Routing + Generated data (not just
  routing). Data word 256 B / 64K words / 16-bit word address; metadata
  46 B (368-bit) 1:1 with data words; 7 virtual queues. Flagged: bit-level
  RG layout is a literal "<insert Routing Data here>" placeholder in both
  specs; endianness stated nowhere; queue-list typo (overview says

  delivery format (what software sees): one 32 B routing-data write, plus
  a second 32 B generated-data write at +32 iff gd_valid — a 64 B slot per
  vector, in two 64K x 64 B SAM regions (OCM vs DRAM per
  acp_mem_target_sel).

## Gotchas & decisions

- The specs' "<insert Routing Data here>" placeholders are real: NO docx
  spec publishes the record packing. The bit-accurate source is the RTL
  repo's design/common/packages/mdata_pkg.sv (verified against the SPM
  RTL that imports it). refmod_exp's copy and asimov-sw Encoding.hs are
  older layouts — regenerate/re-verify both from the RTL package.
- refmod_exp is the reference-model repo, NOT the RTL. Its mdata_pkg.sv
  is STALE (send_to at [2:0], spm_addr[18:3], spm_stride/spm_count —
  all superseded). The RTL repo Positron-AI-TSMC/asimov became
  reachable to this account on 2026-07-10 (gh CLI as jhan-positron);
  fetch files via `gh api repos/Positron-AI-TSMC/asimov/contents/<path>`
  (base64) — plain HTTPS clone of the full repo is heavy (~14.5K files,
  large IP trees).
- Even RTL comments can be stale — the mdata_pkg ACP header comment
  contradicts the cmn_acp_mgr.sv code (which wins: 64 B slots,
  acp_rd_addr as index, per-core base CSRs). Verify against code, not
  comments.
- Metadata is push-only toward software (ACP writes into L2 with cache
  stashing); there is no memory-mapped read path for RG records (debug
  registers aside). Writing records via the CMN AXI ports IS possible and
  is how ARM cores inject notifications.
- Virtual queues store 16-bit ADDRESSES of records, not records; bypass
  mode (default, >= 8 credits) skips rg_mem and the virtual queue
  entirely. A queue returns to bypass only once its virtual queue fully
  drains.
- docx tooling on this machine: no pandoc/python/node — unpack with
  unzip + perl, read diagrams from word/media.
- Plain git with the stored Windows credential can clone/fetch/push any
  reachable positron-ai repo (blobless clones make org-scale sweeps
  cheap), but CANNOT enumerate org repos or search code — that needs the
  gh CLI or the GitHub connector; and extracting the raw token from the
  credential store is (correctly) permission-blocked.
- claude.ai/Code-tab connectors attach at session START only — a
  mid-session OAuth grant requires a new session to take effect.
- CMN_uArch_Spec.docx only appeared in the docs folder on 2026-07-07 —
  page scope on multi-day spec studies should be re-checked against doc
