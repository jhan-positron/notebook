## Gotchas & decisions
- **`core-power enable` silently wipes CLOS3's boot clamp config** (800-2700), and `turbo-freq enable --auto` wipes *all four* CLOS configs and rewrites the whole assoc partition — v2 re-pins CLOS0/CLOS3 on every apply and never uses `core-power enable`.
- **SST state does not survive reboot** (BIOS reprograms the PCT partition every boot) — re-run `flat_freq_apply` after any reboot.
- **sudoers/non-interactive quirk**: on 3bda only `/opt/intel-speed-select/intel-speed-select` has NOPASSWD; on 3af6 only the workspace build does. v2 auto-detects; `FLAT_FREQ_ISST` env var overrides. The user's original wrapper's sudoers precheck fails non-interactively on 3bda.
- **Control-plane pinning**: `--min 800 --max 2000` starves to ~0.8 GHz under contention; must use `--min 2000 --max 2000`.
- CLOS1/2 "unlimited" restore writes `--max 25500` — a `Max Turbo frequency` sentinel, byte-identical to boot per readback, but a sentinel write, not a no-op.
- Aggregate math that killed hetero: 4x4.4 + 140x2.7 = 395.6 GHz-cores vs 144x3.27 ≈ 470 — clipping the rest to 2700 costs far more than the boost gains; the only theoretical win (16x4400 + 64x4100, +1.5%) needs both the pcode rule broken and a BIOS C6 revert, and E2 showed the rule holds.
- The 17:46:32 `flat_freq_apply` collision is why RUN1's P2-P5 measured the user's flat state; the TF-on addendum (RUN2) supplies the intended data — don't be confused by RUN1's raw captures.
- Concurrent-use hazard generally: user and Claude both drive the same machine; check `flat_freq_status` / journal before trusting a measurement window.
