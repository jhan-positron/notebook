What Core-Power Is

intel-speed-select core-power is the Intel Speed Select Technology interface for per core/tile priority. On this host it exposes CLOS classes: each logical CPU can be associated with a CLOS, and each CLOS can carry policy fields such as weight, minimum frequency, maximum frequency, desired frequency, and EPP.

It is best understood as a policy and arbitration mechanism inside a package power domain. It is not a hard per-core frequency setter. A high CLOS can permit a core to run higher and give it priority, while a low CLOS can cap or de-prioritize other cores.

How Core-Power Is Configured

Configuration Flow

  1. Run host preflight: verify /dev/isst_interface, ISST modules, cpufreq governor, and sudo -n turbostat.
  2. Pick a power-domain anchor CPU. This run used CPU24 for the 24-47,168-191 domain.
  3. Capture baseline core-power info, perf-profile info, and existing CLOS associations.
  4. Enable core-power on the anchor CPU.
  5. Configure CLOS0/CLOS1 policy values.
  6. Associate both selected CPUs and their SMT siblings to the intended CLOS.
  7. Measure with turbostat/perf/cpufreq, then restore baseline and read it back.

Working Command Shape

intel-speed-select --cpu 24 core-power info
intel-speed-select --cpu 24 core-power enable --priority 0
intel-speed-select --cpu 24 core-power config --clos 0 --weight 15 --min 3900 --max 4400
intel-speed-select --cpu 24 core-power config --clos 1 --weight 15 --min 800 --max 3900
intel-speed-select --cpu 24-47,168-191 core-power assoc --clos 1
intel-speed-select --cpu 28,172 core-power assoc --clos 0

The final command is an example for CPU28 plus its SMT sibling. The matrix expanded this list for k=1..5 selected workers.

Core-Power And Turbo Frequency

The command-line features are separate: intel-speed-select core-power ... configures CLOS policy and CPU association, while intel-speed-select turbo-freq ... controls Speed Select Turbo Frequency state. There is no syntactic argument dependency where a core-power command takes a turbo-freq argument.

The runtime dependency is real, though. Core-power can only request, permit, cap, or prioritize within the frequency headroom that the package's turbo tables and power budget make available.

Base Config vs Active SST-TF Policy

The base configuration in this run had speed-select-turbo-freq:enabled, so the SST-TF feature was available and normal turbo-ratio tables were active. But base did not enable core-power, did not create high/low CLOS associations, and did not explicitly designate a small set of target CPUs as high priority.

That is why I would categorize base as normal Intel Turbo Boost behavior with the platform's SST-TF capability enabled, not as an active SST-TF high-priority layout. The follow-up active SST-TF experiment deliberately chose target high-priority CPUs through the turbo-freq feature's auto path, verified that low-priority cores were clipped, and then measured the selected high-priority bucket.

Does Turbo-Freq Need To Be Enabled?

For this performance goal, leave turbo capability in the normal enabled baseline. Disabling turbo-freq is a flat-frequency baseline, not a prerequisite for core-power. If the goal is to raise a few selected cores toward 4400 MHz, disabling the turbo mechanism works against the goal.

Core-power is not irrelevant to turbo; it is bounded by turbo. The 2-core Gate A case had turbo headroom and split 4360 versus 3900 MHz. The 24-core matrix had no headroom above the 3900 MHz active-core bucket, so CLOS0 max 4400 did not matter.

Underlying Technologies

  • Intel Speed Select Core Power (SST-CP): exposes per-core/tile priority via CLOS policy and CPU association.
  • Intel Speed Select Turbo Frequency (SST-TF): enables different turbo ratio limits based on core priority. In base, this feature was enabled but no high/low priority CPU layout was explicitly applied. The platform reports an 8 high-priority-core bucket at 4400 MHz; the follow-up active SST-TF layout used that bucket successfully for k=1..5 selected workers.
  • Turbo ratio limits: active-core-count tables define ceilings such as 3 active cores -> 4400 MHz and 24 active cores -> 3900 MHz.
  • HWP/cpufreq/intel_pstate: Linux policy, EPP, and hardware P-state selection participate in the requested and achieved operating point.
  • Power domain: these choices are not isolated to one logical CPU. They are mediated by package/domain limits, power, thermals, and SMT sibling placement.

Caveats

  • clos-max=4400 is an upper permission, not a promise that the core will run at 4400 MHz.
  • Always include SMT siblings in CLOS association so a selected physical core is not split across classes.
  • The default sandbox can hide /dev/isst_interface; host/escalated context is required before declaring ISST unavailable.
  • Test the real privileged measurement command, sudo -n turbostat. Do not use sudo -n true as the project proxy.
  • Restore policy after every run: turbo-freq baseline, core-power disabled, CLOS disabled, all tested CPUs back to CLOS0.
