delphi-3bda · flat-frequency investigation · June 30 – July 19, 2026

The missing decode boost, explained

In early July, removing the CPU frequency clamp appeared to speed up token generation by ~13%. Weeks later we could only measure +5–8% no matter what we tried. This page explains, in plain language, what we found: nothing got slower — the yardstick moved.

The question you asked

Around July 3 we measured the flat-frequency fix (letting worker CPU cores run at ~3.9 GHz instead of the factory-locked 2.7 GHz) and saw two improvements on gpt-oss-120b with 8 users:

  • Prompt processing ("prefill" / "parse"): about +17%
  • Token generation ("decode" / "generate"): about +13%

Recent tests kept showing the same prefill gain but only +5–8% for generation. You asked: what happened to the rest, and how do we get it back?

The answer

The prefill boost never went anywhere — it is +16–18% in every test, every era, every harness. Rock solid.

The generation boost didn't shrink because the machine got slower. It shrank because the slow side of the comparison got faster. At full speed, the machine generates 103–105 tokens/s/user today versus 102.6 in the July 3 measurement — the same or slightly better. But when we lock the cores back down to 2.7 GHz to measure the "before" number, today's machine does 95–98 tokens/s where the July machine did only 90.6. A boost is a ratio: same top speed ÷ faster baseline = smaller percentage.

So there is nothing to bring back. Peak performance never regressed. The old +13% number was computed against an unusually slow clamped baseline — a quirk of the machine's state at that time, which was erased by the July 17–18 maintenance and cannot be recreated (we tried every software explanation; see the elimination table below).

Going forward, the honest expectation for the flat-frequency fix at the standard 1024-token benchmark is +16–18% prefill and +5–8% generation — and both grow with prompt length (at 4096 tokens: +26.6% prefill, +13.4% generation; see the prompt-length section). The nightly CI numbers were accurate all along.

Jargon decoder

The short names used in the README and handoff:

clamp / clamped
The factory boot setting that locks most CPU cores to 2.7 GHz. Our "before" condition.
flat-freq fix
Our fix that releases the worker cores to run ~3.9–4.0 GHz. The "after" condition.
serving / serving path
THE definition to hold onto (used consistently below): the production route a request takes — over HTTP into the rinzler server process, which runs the engine and streams tokens back. Customers use this path; so does the CI test. The opposite is "engine-direct": running the engine straight from a dev tool with no HTTP at all.
checkerboard / runtron
The team's benchmark harness. It drives runtron, a developer command-line tool that runs the inference engine directly — it does NOT use the serving path.
CI / talos
The nightly automated test. It sends requests through the serving path, like customers do.
front-end cores
CPU cores 1, 2, 73, 74 (+ their HT siblings) — where the serving process keeps its non-worker threads (the HTTP/streaming threads, and — as the July 20 trace revealed — one important engine thread per instance). Clamped at 2.7 GHz in the production frequency shape.
SSE (Server-Sent Events)
The HTTP streaming mechanism behind stream=true: the connection stays open and the server pushes each generated token to the client as a small data: chunk. Not a dedicated thread — the delivery work runs on the HTTP framework's (Drogon's) shared event-loop threads, which multiplex many client connections. Measured cost: ~46 µs per token — three orders of magnitude too small to explain any serving cost, which is what ruled HTTP/streaming out.
0bf
Nickname for engine build 198650bf — the older code baseline, from before the new core-layout change. Same lineage as the July 3 measurement.
aa8
Nickname for engine build 29924aa8 — newer code that includes the improved die-aware core layout (PR-3070).
RTM / TSX
An Intel CPU feature the engine uses so that worker threads waiting for the FPGA can "park" efficiently, consuming almost nothing. Without it, waiting threads spin in a busy loop, burning CPU cycles. It requires a kernel boot flag (tsx=on); 3bda ran without it until the July 13 reboot.
TRON_NO_RTM
An engine switch that forces the old busy-spin waiting even when RTM is available. We used it as a control: "recreate June's waiting behavior on today's machine."
arena allocator
A newer, faster memory allocator inside the engine (pos_heap_v2). It was OFF in the June builds and became the default afterwards — another "what changed" suspect we tested.
SYSTEM_CONFIG
A shell variable in your dotfiles that silently overrides which FPGA slots the engine uses. It broke a weekend of test runs after the maintenance reboot (now durably fixed with unset).

Does prompt length change the boost?

Yes — confirmed on the clean host (July 20). All the headline numbers above use 1024-token prompts, so "+5–8% generation" is a 1024-token statement. The clean length sweep (aa8 build, RTM on, clean environment, both instances healthy in all 18 launches, 2 reps, per-cell power) shows the boost growing steadily with prompt length:

So the June-era observation was directionally right all along: its slope was real, only its level was inflated by the old host state (compare the last two columns — each June point reads ~3–5 points above the clean one). The mechanism is the one this page keeps returning to: during generation the attention math runs on the CPU and its cost grows with context, so longer prompts make generation more clock-sensitive. Memory power tells the same story (RAM watts rise from ~50 W at 256 tokens to ~66 W at 4096).

Practical takeaway: the flat-frequency fix is worth more the longer the context — +7% generation at 1024-token prompts, +13.4% generation and +26.6% prefill at 4096 tokens. Long-context workloads benefit substantially more than the standard benchmark suggests.

A model that predicts generation performance

The length sweep supports a simple two-part model of where each generated token's time goes — call it the floor + slope model — accurate enough to use for tuning predictions. Fitting all 16 measurements:

time per token (ms) ≈ FLOOR + SLOPE × avg_context (k tokens)

fast clocks:  8.27 ms + 1.05 µs/token

clamped clocks:  8.36 ms + 1.37 µs/token

avg_context ≈ prompt length + generated/2 (context grows while generating). TPS = 1000 ÷ time. Note: this sweep generated as many tokens as the prompt in every cell (gen = prompt), so its avg_context works out to ~1.5×prompt — a property of the sweep design, not of the formula. CI's config (p1024, 1536 generated, capture at tokens 896-1024) samples at ~2k context, for example.

The floor is the FPGA matmuls plus waiting, plus the orchestrator's I/O-bound job-prep (descriptor/doorbell round-trips — see the iteration diagram in the Effect 2 section). It is context-independent and independent of CPU core clocks: the two fits land within 1% of each other (8.27 vs 8.36 ms) across a 48% core-clock change — and that residual 1% is real signal, not noise: it matches the orchestrator's small clock-bound serial slice, the same ~0.1 ms/token behind the +1.2% front-end un-clamp. Two things the floor is not immune to (measured Jul 20-21): the mesh/uncore clock — capping it 2.5→1.6 GHz adds 6-7% to the floor's I/O component — and the provisioning draw (see "Problems to investigate further"). The slope is the CPU attention pass over the KV cache — it is the only part that responds to core clocks, and only partially (×1.30 faster for ×1.48 clock: attention is partly memory-bandwidth-bound, corroborated by RAM power rising ~50→66 W across the sweep).

How well does it predict?

  • In-range predictions land within ±3.6% of measurement at every length, both clock settings.
  • Leave-one-out test (fit on three lengths, predict the fourth): interior points within ±4.5%; extrapolated endpoints within ±10%.
  • The model's derived boost curve matches the measured one within ~1.5 points at every length (predicted +2.4 / +5.8 / +9.3 / +13.9% vs measured +0.9 / +7.0 / +10.1 / +13.4%).
  • Structure check on old data: the June 24h sweep stays linear in context out to p8192 (~12k ctx), so the two-part form holds well beyond the fitted range (with era-specific constants).
  • Accuracy limit (added Jul 21): any single run carries the ±3-4% provisioning lottery (see "Problems to investigate further") — the model predicts means; part of the leave-one-out scatter above is that lottery, not model error.

Using it to rank tuning ideas before running them

Because the two parts respond to different levers, the model predicts what a tuning idea is worth before anyone builds it:

  • Faster CPU clocks / more worker cores only shrink the blue part → worth +7% at 1k prompts but +13–19% at 4–8k. (This is the flat-freq result, predicted.)
  • Cutting the floor by 1 ms → ~+11% at 1k prompts, ~+7% at 4k — helps short contexts most, helps everywhere. Measured leads (Jul 20-21): the orchestrator's ~0.2-0.3 ms latency-bound serial head, and simply keeping the mesh/uncore uncapped (a cap silently adds 6-7% to the floor).
  • Halving attention cost (kernel work, bandwidth) → ~+9% at 1k but ~+28% at 4k — the long-context lever.
  • Extrapolating the curve: at p8192 the model predicts ~47 tok/s fast / ~40 clamped (boost ~+19%) — testable whenever a long-context config lands.

Why numbers through the serving path read a few percent lower

Extending the model to the serving path answers "why does CI decode read so much lower than the engine-direct benchmark?" precisely. Two effects stack, and neither is what it first looks like.

Effect 1 — the metrics sample different parts of the same request. Each token's cost rises during a request as context grows (token #1 attends over 1024 tokens; token #1024 attends over ~2048). The engine benchmark averages every token of the request; the CI metric deliberately measures only tokens 896–1024 — the slowest portion of the request (the last 128 tokens, where context is largest). This is straight from the CI code (repo positron-ai/systems_test, verified on main):

So the two harnesses divide different things: CI reports 128 tokens ÷ the time between token 896 and 1024 arriving (steady-state by design); the benchmark reports ~1024 tokens ÷ the whole generation time — an average over the entire tilted line:

Effect 2 — an engine thread living on the clamped front-end cores (mechanism traced by the July 20 perfetto A/B). It is not the streaming: token delivery costs only 46 µs/token, and in 60-second scheduler traces of both arms, no engine thread was ever woken by a front-end thread — zero events across 47,472 tokens. The real mechanism: each engine instance's work_queue thread — the forward-pass orchestrator that prepares and enqueues every FPGA matmul job, writes descriptors, and does MoE routing — is confined by the process affinity to front-end cores 1 and 73, at the clamped 2.7 GHz. Important qualifier: this thread occupies its core ~99% of the time, but ~70% of that occupancy is spin-waiting on FPGA completions — the real work is ~2.9 ms per ~9.9 ms iteration (~29%). Of that 2.9 ms, ~2.05 ms is I/O-bound (register/descriptor round-trips to the FPGA — fixed time, insensitive to CPU clock) and ~0.84 ms is genuinely clock-bound. Clocking the core 2.7→3.9 shrinks the clock-bound part; the slice of it the FPGA was actually waiting on is ~0.12 ms per token — which is exactly what +1.2% of a 9.9 ms token means (P4.3 measured that +1.2% directly, three interleaved pairs). So what looked like "serving overhead" is really an engine thread that happens to live on the front-end cores.

The kitchen version. The FPGA is the oven; the 112 workers are line cooks at fast stations; work_queue is the expediter who writes every order ticket and loads every oven tray — and the oven can't start a tray he hasn't prepared. His desk sits in the slow corner of the kitchen (a clamped core). Most of his 2.9 ms per cake is filling in forms with slow hardware (goes at the hardware's pace no matter how fast he is); ~0.84 ms is his own writing speed. Move his desk to a fast corner and the tray-prep the oven was waiting on shrinks ~0.12 ms per cake: +1.2%. The waiters (SSE) just carry finished plates — 46 µs, never once holding up the kitchen. And nobody in this kitchen rings a bell when handing work over — they watch each other's counters (spin-waiting) — which is why the OS's "who woke whom" ledger was blank and it took stopwatch-style profiling to find the expediter.

Why prefill gains more than generation — expected, not a bug

During prefill, the CPU does heavy number-crunching (attention over the whole prompt). Faster clocks → proportionally faster work. Hence +16–18% everywhere, always.

During generation, the FPGA does the big matrix math and the CPU spends roughly 85% of each token's time waiting for it. Speeding up a CPU that's mostly waiting buys little — the measured +5–8% is exactly what that arithmetic predicts. The June +13% was the outlier that needed explaining, and its explanation turned out to be a slow baseline, not a fast peak.

Does the CI test under-stress the system? — checked, and no

A fair challenge raised during the investigation: maybe the CI test sends too few (or repeated) prompts, and its rounds idle while waiting for the slowest request — so a "looping" tool would stress the system harder. We verified this from the CI code and five nights of recorded production data. The code says otherwise:

# testlib/tps.py:142-145 — a DIFFERENT prompt for every user in every round
prompt = wkr.prompt_generator.generate(
    prompt_length=wkr.config.prompt_length,
    seed=round_index * wkr.config.n_users + index,   # 8 users x 10 rounds = 80 unique seeds
)

# testlib/prompt.py — each seed picks a different corpus conversation,
# plus a seed-derived timestamp making every prompt unique even beyond that
convo = self.convos[seed % len(self.convos)]['conversations']   # 1000-convo corpus
system_prompt = {'from': 'system',
                 'value': f'[TIME: {fake_time}] You are a helpful assistant.'}

# testlib/tps.py:374 — every request decodes the FULL 1536 tokens, no early stop
extra_body={ "ignore_eos": True },

# testlib/tps.py:76,356 — 8 concurrent users, staggered 0.1 s
stagger_delay: float = float(os.environ.get('STAGGER_DELAY', 0.1))
time.sleep(user_index * config.stagger_delay)

So one CI run = 80 unique cold prompts through the full serving path, each forced to generate all 1536 tokens, at 8-way concurrency. The barrier rounds are real (a round ends when its last request finishes) — but because prompt and generation lengths are uniform, all 8 users finish within ~1–2 s of a ~17–19 s round: ~90% of every round runs at full 8-user load, and the measured window samples only that saturated steady state. The recorded data agrees: across five production nights (400 requests), zero requests show cache-hit signatures — TTFT median 1.17–1.33 s, never under 300 ms, cached tokens only the ~71-token template preamble.

The comparison tool, meanwhile, stresses less, not more. "tronperf" is a profiling wrapper around runtron whose recorded use sends 8 fixed Moby Dick chapters, once, with no loop — and its only loop option replays the same prompts, which the engine serves from the prefix cache:

params.prompt_files.push_back("/scratch/prompts/md_ch" + std::to_string(chapter) + ".txt");

# run-tronperf.sh (the recorded invocation) — one-shot batch, no loop flags at all

The refined challenge: round-boundary idle — half right, and the right half is already queued

A sharper version of the concern: within a round, when a user's request finishes, the harness does not immediately send that user's next prompt — everyone idles until the slowest request completes, then the next round starts together. A "no-wait" design would keep 8 requests in flight at all times.

The critique of the design is correct — and with variable-length real traffic it would matter a lot. Two facts complete the picture, though:

  • The recorded numbers are immune, by construction. The config forces 1536 generated tokens while the capture window ends at token 1024 (see the perf.py snippet above) — the earliest any user can finish is token 1536, so every measurement window closes while all 8 users are still generating. The idle tail happens entirely after the last measurement of each round: it wastes wall-clock time and weakens sustained-load realism, but cannot touch a recorded decode number.
  • "No-wait" is not how the comparison tool ran. Every recorded runtron/tronperf invocation is a one-shot batch — all prompts at t=0, run to completion, exit: a single barrier round, then nothing. (runtron does have an unused closed-loop flag, --concurrency — the capability exists, but no run behind any of our numbers used it.)

One kernel of the challenge was right and got adopted: the CI's 80-prompt set is byte-identical every night (the seeds have no run-to-run variation), and its prefill metric never corrects for cache hits — harmless tonight only because the nightly rebuilds the engine and cycles 8 other models first. Back-to-back same-config reruns would silently report ~10× fake prefill. The hardening (date-derived seed rotation, cache-aware prefill with a fail-loud gate) is in the test-harness PR queue.

Note: the official CI harness needs an update. Much of this investigation ran on purpose-built tooling that exceeds the official harness in specific ways — loadgen.py (a no-wait closed-loop client with unique per-request prompts and a think-time option, written during the investigation — it is not part of CI), a patched harness clone (seed rotation, long-prompt support), and per-run provenance capture (power, host snapshot, cache-coldness, engine banners). The official systems_test should absorb these; the queued PR covers: ① a no-wait stress config (continuous arrivals, mixed lengths) alongside the steady-state config; ② date-derived prompt-seed rotation; ③ cache-aware prefill with a fail-loud gate; ④ longer-prompt configs (2k now, 4k behind a corpus fix); ⑤ a clamped-arm canary and trailing-median regression bands to replace the loose static goals (the last green nightly passed at 178% of goal — a 6% drift would never alarm); ⑥ per-run provenance/power capture. Branch jhan/seed-offset-long-prompts already carries the first pieces.

Useful things found along the way

  • The nightly CI is trustworthy — it sends 80 unique prompts per run through the real serving path (verified in code and in five nights of recorded data), and its numbers matched every controlled experiment.
  • The new core layout (aa8 / PR-3070) is genuinely better: +3.7% gen and +14% prefill over the legacy layout at identical clocks, through the real serving path.
  • The SYSTEM_CONFIG landmine in the dotfiles silently re-arms on every reboot and broke 48 straight test launches after the maintenance — now durably fixed with unset (and orchestrators scrub it too).
  • API keys were exported world-readable in ~/.bashrc on the shared filer — rotate them, keep them in a chmod 600 file.
  • Hardening for the test harness (queued as a PR): rotate prompt seeds nightly, make the prefill metric cache-aware, and add a production-shaped stress config.

Problems to investigate further

1. The provisioning lottery — run-to-run noise with no visible cause

Six fresh provisionings of the identical serving config, measured back-to-back on July 21 (one 240 s load block each, nothing else changed): decode varied 96.84 → 103.66 tok/s/user — a 6.9% max−min spread (CV 2.4%). The obvious suspect was instance placement (which units/FPGA pairs/cores each instance lands on), but the assignment fingerprints were byte-identical across all six draws — same units, same devices, same cores, same NUMA nodes. Whatever varies is invisible to the command line: candidates are hugepage physical layout, weight placement in memory, or FPGA-side state.

Six provisioning draws, identical config & assignment

The practical rule until this is solved: a single night's CI delta under ~5% is weather, not a regression — each nightly rides one provisioning draw. Regression calls need trailing bands over several nights (the planned variance week, plus per-run provenance capture, turns this rule into thresholds).

Leads for the follow-up: capture deeper per-draw state (hugepage physical maps, weight NUMA placement, FPGA link/state readouts) and correlate against the draw's speed; a week of seed-rotated nightlies gives the night-to-night distribution for free once the harness PR lands.

2. Pipeline handoff dead time — ~3.4 ms per token, the biggest known headroom (measured Jul 22)

Measured from tron's own track events (ab36: 600k flow-paired launch→collect intervals, 8 devices, steady 8-user serving at 98.6 tok/s/user): the FPGA pipeline is occupied only ~41% of wall clock (40.0–42.0% across devices), CPU attention ~21%, the orchestrator's serial head ~4% — and ~34% (~3.4 ms per token) is stage-handoff latency where neither the CPU nor the FPGA computes: DMA transfers, mailbox waits, descriptor round-trips, thread-relay lag, paid ~11,000 times per second per device in ~16 µs slices (p99 ~1 ms). This supersedes the earlier ~0.2–0.3 ms estimate, which was only the orchestrator-side residue visible to the phase profile. It is latency, not compute — CPU clocks don't touch it (+48% clock → −1%), the mesh clock does (cap costs 6–7%) — so the levers are structural: deeper per-device job queues, batched/coalesced descriptor and DMA traffic, fewer relay hops per layer. Refinement (Jul 22, worker-attention overlay on the same trace): much of any one device's idle is covered by other engines computing (workers, other devices) — at the whole-instance level something computes 86–88% of wall, so the truly global dead time is 12–14% (~1.2–1.4 ms/token; +~14% if eliminated), and the larger structural prize is deeper cross-layer/cross-user overlap so each engine's individual idleness (every FPGA ≤~28–41% busy) shrinks. Still the strongest candidate for the tron team's attention from this whole investigation.

3. Worker load imbalance inside an instance

The sampled worker threads within one instance carried unequal duty — 29.1 / 19.9 / 17.5 / 16.6% — one core doing ~1.5× the work of its peers. Not obviously costing throughput today (workers spin-wait most of the iteration), but an uneven partition is a latent bottleneck as models or batch shapes change. Also one for the tron team.
