System Performance Research Workflow Template

A reusable instruction pattern for long-running CPU/system-performance investigations: begin with PAL/Claude consensus, collect real CPU-work evidence, correlate traces with counters, gate every step, and feed the lesson back into the performance knowledge base and future AI inputs.

Start: PAL + claude-opus-4-8 Evidence: Perfetto + perf + turbostat Discipline: consensus before proceed Output: distilled knowledge base

Reusable Prompt Opening

The workflow starts with an explicit consensus contract. Keep this as the first line of future system-performance prompts, then append the specific experiment.

Use PAL to deep work with claude-opus-4-8 on the following system performance investigation.

For every step:
- you and Claude must reach consensus on the step result before moving to the next step;
- if evidence surprises either of you, stop, update the plan, and converge again;
- record enough handoff state that the run can resume after interruption.

Do not rely on Linux scheduled CPU time alone. Collect real CPU-work counters,
correlate them with Perfetto thread/core residency, and distill accepted findings
back into the system performance knowledge base.

Core abstraction

Turn every performance claim into a gated evidence claim:

trace location + CPU counters + frequency/power + peer consensus

Only after all four agree should the result become a recommendation or knowledge-base entry.

3

Instrument

Execute Gates

Run one gate at a time: baseline, candidate, restore, drift/sentinel, analysis, and final recommendation.

Correlate

Claim Quality

Call a thread heavy only when residency, retired instructions, cycles, IPC, and wait-path samples support the same story.

Measurement Required Data Why It Is Gated
Scheduled residency Perfetto TID/thread/core timeline Shows placement and pinning, but can be fooled by polling or wait loops.
Real CPU work Per-CPU and best-effort per-thread perf stat: cycles, instructions, ref-cycles, cpu-clock, task-clock, IPC Separates useful retired work from simply being scheduled.
Wait/spin attribution Low-rate perf record with symbols Identifies mwait, pause, spin, or polling functions that inflate Linux busy views.
Frequency and power turbostat: Busy%, Avg_MHz, Avg_Bzy_MHz, C-states, CPU Package Watts Confirms hardware-side active frequency, idle behavior, and power tradeoff.
TSX/RTM state rtm_retired.start, rtm_retired.commit, rtm_retired.aborted, CPU flags / CPUID evidence Answers whether transactional execution is active or just a red herring.

Consensus-Enforced Gate Procedure

G0

Preflight Gate

Confirm model/config, host state, permissions, device nodes, tool versions, expected CPU sets, output folders, and resume plan.

Consensus required
G1

Baseline Gate

Run baseline with full instrumentation. Proceed only if return codes, error rate, policy readback, perf counters, trace capture, and turbostat are valid.

Consensus required
G2

Candidate / Experiment Gate

Apply one change, verify readback, run the same workload, and compare shape before claiming performance movement.

Consensus required
G3

Restore / Drift Gate

Return to base, repeat sentinel runs, and decide whether deltas exceed run-to-run drift and measurement noise.

Consensus required
G4

Analysis Gate

Correlate Perfetto and perf evidence. Separate scheduled time, retired work, wait/spin samples, RTM/TSX counters, frequency, and watts.

Consensus required
G5

Execution Discipline For Long Runs

Prepare For Hours, Not Minutes

Write a concrete matrix, expected duration, output paths, stop conditions, rollback commands, and sentinel checks before the first live run.

  • Create run directory up front.
  • Save exact commands before execution.
  • Log host state and tool versions.

Resume Robustly

After every milestone, write a short handoff containing completed gates, current host policy, valid artifacts, failed artifacts, and the next safe command.

  • Never assume partial data is valid.
  • Mark invalid attempts explicitly.
  • Keep base-restore commands nearby.

Consensus Before Motion

Each transition is a contract: Codex and Claude agree on the previous result and the next step. If they disagree, pause the plan and resolve the evidence gap.

  • No silent plan drift.
  • No skipping gates after a surprise.
  • No final recommendation without drift context.

Use PAL to deep work with claude-opus-4-8 on the following system performance research task:

Objective:
- [state the workload, host, config, and decision needed]

Evidence requirements:
- Collect Perfetto/tronperf thread-name/TID/core residency.
- Collect per-CPU perf stat for expected pinned CPUs:
  cycles, instructions, ref-cycles, cpu-clock, task-clock,
  rtm_retired.start, rtm_retired.commit, rtm_retired.aborted.
- Collect best-effort per-thread perf stat after long-lived workers exist.
- Collect low-rate perf record with symbols to identify mwait/pause/spin paths.
- Collect turbostat with Busy%, Avg_MHz, Avg_Bzy_MHz, C-states, and CPU Package Watts.

Correlation rule:
- Do not call a thread heavy from scheduled residency alone.
- Correlate Perfetto TID/core timeline with retired instructions, cycles, IPC,
  wait-path samples, RTM/TSX counters, frequency, and power.

Execution gates:
- G0 preflight: plan, tools, permissions, host state, output paths, resume plan.
- G1 baseline: run and validate all artifacts.
- G2 candidate: apply one change, verify readback, run and validate.
- G3 restore/sentinel: return to base and measure drift.
- G4 analysis: compare performance, frequency, power, and real CPU work.
- G5 distill: update final report, visual, and distilled knowledge base.

Consensus discipline:
- Each step requires Codex/Claude consensus before proceeding.
- If a surprise appears, stop, update the plan, and converge before continuing.

Long-run discipline:
- Prepare for long runtime.
- Write milestone handoffs so execution can resume robustly.
- Keep rollback/base-restore commands visible.

Feedback:
- Add a final feedback section: good/bad/ugly of the instruction and exact wording
  to improve the next input to AI.
