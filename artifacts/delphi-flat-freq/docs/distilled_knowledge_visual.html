Intel SST Distilled Knowledge

Common operating model and CI performance lessons for delphi-3af6, Intel Xeon 6962P. Generated from the flat_freq reports plus later thread-shape and core-power handoffs.

One-Screen Verdict

flat_freq Policy
Disable SST-TF

Configure all six anchors with turbo-freq disabled and core-power disabled.

Performance Law
Frequency Sensitive Wins

flat_freq wins when the workload can turn all-domain ~3.9 GHz into useful work.

Power Caveat
Throughput, Not Watts

CPU package watts usually rise. flat_freq is not an energy-per-token policy.

Use Avg_Bzy_MHz for average busy-clock frequency. Some artifacts use different casing, but the durable convention is Avg_Bzy_MHz.

Two Knobs, Two Different Jobs

turbo-freq / SST-TF

Priority-based turbo-ratio policy. Base enables it, creating a high-priority/low-priority split.

Base effect: one active domain can show p90/max near 4.4 GHz while other active domains are clipped around 2.7 GHz.

core-power / SST-CP / CLOS

CLOS priority tagging. Define a few priority classes, then associate CPUs/cores to a class.

Important: flat_freq keeps core-power disabled; it does not use CLOS to set per-core priority.

Does core-power mean we can set one individual core's priority?

Mostly yes, but not as arbitrary per-core GHz control. core-power uses CLOS classes: first define a small number of classes, then associate a CPU/core with one of those classes. The class can carry min/max frequency hints and, in proportional mode, a proportional-priority value; the CPU association is the per-core part.

LayerWhat is configuredExample meaning
CLOS class--clos, --weight, --min, --maxDefines a class. Ordered mode and proportional mode use different priority semantics.
CPU association--cpu <n> core-power assoc --clos <k>CPU 28 belongs to CLOS 0
Intel tool source says core-power defines per-core/tile priority. The IOCTL header says priority_type is ordered/proportional, clos is the CLOS ID assigned to a logical CPU, and prop_prio is proportional priority from 0-15. The parser/help maps --priority 0 to proportional mode and --priority 1 to ordered mode. This is source-derived; host/escalated readback confirms the current platform reports priority-type:proportional, but CLOS mutation and weight direction are not measured yet.

Representative command template for a future mechanism probe. The effect/direction of --weight must be validated by readback and measurement before making a tuning claim:

BIN=/home/jhan/workspace/intel-vs-amd/speed-select/workspace/flat_freq/20260623_codex_preserved_tmp/intel-speed-select-v6.8/intel-speed-select

# Inspect core-power state on a domain anchor.
sudo $BIN --cpu 24 core-power info

# Enable core-power/CLOS in proportional-priority mode.
sudo $BIN --cpu 24 core-power enable --priority 0

# Define CLOS 0 for the target CPU in the probe.
sudo $BIN --cpu 24 core-power config --clos 0 --weight 15 --min 3900 --max 25500

# Define CLOS 1 for the comparison CPUs.
sudo $BIN --cpu 24 core-power config --clos 1 --weight 0 --min 3900 --max 25500

# Associate one CPU/core with CLOS 0.
sudo $BIN --cpu 28 core-power assoc --clos 0

# Verify association and class config.
sudo $BIN --cpu 28 core-power get-assoc
sudo $BIN --cpu 24 core-power get-config --clos 0

Thread And Resource Shape

Current CI shapes intentionally do not use all 144 physical cores or 288 logical CPUs. Runtron instances use device-facing cores, a main/work-queue core, and app-worker cores.

Terminology

dev-cores are TX/RX runtime cores. app-cores are runtime/model worker cores. platform_cores are reserved for platform/sidecar work.

GPT-OSS / MoE Lessons

Model Shape
36 x 128

36 layers, 128 routed experts, top-k 4.

Routing Shape
~13 experts

Decode touched about 13 distinct experts per 4-user pass; prefill about 101.

Randomized Experts
Stress Mode

TRON_RANDOMIZE_EXPERTS=defined is a validity/stress condition, not a production-routing claim.

The active expert footprint is a logical/cold-stream estimate, not measured DRAM traffic. The key inference is that cache/reuse leaves a frequency-sensitive fraction; that fraction grows with prompt length, explaining the GPT-OSS decode slope.

Metric Dictionary

Avg_Bzy_MHz

Average MHz while CPU was not idle. Summary row is across all CPUs and the interval.

CPU Package Watts

Summary-row package watts across both sockets unless split per socket.

Busy%

Hardware activity ratio, not proof of useful work. Pair with C-states and watts.

TSX/MWAIT

RTM counters were zero. Linux thread CPU% can overstate spin/wait behavior.

Decision Rule And Open Questions

Use flat_freq When

CI is throughput-oriented, many domains participate, prompts are long, or GPT-OSS/MoE and parse-heavy work matter.

Be Careful When

Work is single-stream, lightly threaded, energy-per-token focused, or dense-model decode is known to be bandwidth floored.
