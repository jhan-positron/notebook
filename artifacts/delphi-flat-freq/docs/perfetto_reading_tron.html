Reading tron in perfetto: is it CPU-bound or FPGA-bound?

A guide through the trace shapes that mislead, written after three perfetto captures and a perf phase profile of gpt-oss-120b tp4 serving on delphi-3bda (July 2026). Short answer (updated Jul 22 with the measured FPGA lane, snippet 7): at the CI operating point, decode is pipeline-latency-bound — FPGA compute ~41% of wall, CPU attention ~21%, and ~34% is stage-handoff latency where neither side computes; the same system is genuinely CPU-bound in other regimes (long context, prefill, small models). Naive trace reading misleads here for two reasons this page demonstrates — and its own first draft fell for one of them (snippet 7's correction).

Snippet 1 — the trap that makes tron look CPU-bound: spin-wait reads as "Running"

What the thread-state track shows for the orchestrator thread

schematic recreation of the sched track for 1/?-work_queue under steady 8-user decode

Analysis. tron's threads never sleep while waiting for the FPGA — they spin (mwaitx/tpause). To the scheduler, and therefore to perfetto's thread-state track, a spinning thread is indistinguishable from a computing thread: solid green "Running". Naively summing Running time says "the CPUs are ~100% busy — CPU-bound!" when the measured real-work duty of this thread is 29%. Any CPU-bound conclusion drawn from Running-state occupancy in a tron trace is unreliable. The same applies to the 112 worker threads (~99% Running, ~21% real work in attention).

Check it in your own trace: you can't, from sched data alone — that's the point. You need an instruction-rate signal: perf stat -t <tid> under load, or a sampled profile (perf record -t <tid>) and look at the callchain mix — spin loops show as a tight wait/poll frame.

Snippet 2 — the second trap: the FPGA has no track

What perfetto shows vs what the hardware is doing

the FPGA appears in a CPU trace only as the absence of CPU activity — and tron never yields, so even that absence is hidden inside "Running"

Analysis. Perfetto traces the CPU. The FPGA shows up only if a driver emits track events, which tron's doesn't. So "the FPGA had a lot of wait time" cannot be read directly from these traces — it has to be inferred, usually from gaps in CPU activity. But trap 1 means there are no visible gaps: everything spins. The CPU-side serial head the FPGA waits on at iteration start is ~0.4–0.5 ms per token — real, and it's the +1.2% the front-end un-clamp recovers. (Correction, Jul 22: an earlier version of this paragraph inferred the FPGA computes "for most of the iteration." The measured lane in snippet 7 shows total FPGA occupancy is ~41% — the serial head is only one of its idle modes; most idle is distributed per-layer handoff latency. The inference trap this snippet warns about caught this page's own author.)

Where the 0.4–0.5 ms comes from — deliberately NOT perfetto. Two independent non-trace sources agree: (1) clock-toggle arithmetic — un-clamping the serial slice's core 2.7→3.9 GHz bought +1.2% = 0.119 ms/token; a slice S clocked ×1.444 faster saves S×(1−2.7/3.9) = 0.308·S, so S ≈ 0.119/0.308 ≈ 0.39 ms — the FPGA must have been waiting on it, or speeding it up couldn't move the token rate; and (2) the perf phase profile (397k samples folded over thousands of iterations): serial head ~0.4–0.5 ms, resolution-limited below ~0.4 ms by ±1 ms period jitter. The perfetto captures contributed the architecture (which thread, its affinity, the 46 µs SSE cost) — not this number. That division of labor is the point of snippet 6.

Snippet 3 — what the correct decode shape looks like (measured)

One decode iteration, real-work view (spin removed)

this is what remains after separating work from spin — the shape the wall-clock-lanes diagram draws

Analysis. When we separated work from spin (perf sampling at ~4 kHz per thread, folded over thousands of iterations), the orchestrator's work turned out to be spread across the whole iteration and interleaved with FPGA compute — the engine already pipelines job-prep under execution. If tron were CPU-bound at this operating point, the FPGA lane would show long idle stretches whenever the blue work runs; instead they overlap ~88% of the time.

What is measured vs drawn in the worker lane. Measured: attention totals ~2.1 ms per iteration (21% duty) and is phase-structured — anti-phased with the orchestrator's work, which is what defines the quiet zone. NOT measured: the burst count and boundaries — the profile gives ~40 samples per iteration per thread, enough for coarse phase structure only. The two solid green blocks are a legibility simplification; architecturally the reality is almost certainly many per-layer bursts (a decoder alternates FPGA matmuls and CPU attention every layer). Do not expect literally two green slabs in a trace — and per trap 1, you will not see this structure in sched tracks at all, only in a work/spin-separated profile.

How "FPGA computing" is inferred, exactly. No event says "FPGA busy" — the inference is progress accounting plus elimination: (1) token timestamps show steady ~9.9 ms progress through every phase of the capture; (2) the compute partition is known — matmuls/MoE live only on the FPGA, the CPU contributes attention and orchestration, there is no third party; (3) the folded profile shows a ~55% stretch of the cycle where neither orchestrator work nor attention is dense, yet progress continues — by elimination, the FPGA is computing there; (4) the orchestrator's own call chains (enqueue_hardware_matmul_job, write_tx_rx_descriptors, hw_prepare) bracket those regions on the submission side. Cross-check: +48% CPU clock moved token time 1% — nothing CPU-side is pacing the quiet zone. The definitive upgrade would be tron emitting enqueue/complete timestamps (a userspace perfetto track): the FPGA lane becomes measured instead of inferred, per layer.

SUPERSEDED (Jul 22): that upgrade turned out to already exist in tron, and the measured lane (snippet 7) revealed the elimination step above was over-attributing: step (3)'s "by elimination, the FPGA is computing there" is only ~41% true — the rest of the CPU-quiet time is stage-handoff latency where neither side computes. Steps (1), (2) and (4) stand; the lesson is that elimination cannot distinguish "the other device is working" from "everyone is waiting on a wire".

Snippet 4 — why sched-based reasoning fails completely here: zero wakeups

Typical pipeline app vs tron, wakeup-arrow view

perfetto's wakeup arrows are how you normally trace producer→consumer handoffs

Analysis. Because handoffs happen by watching shared memory (spin), the scheduler never mediates them, so perfetto shows no wakeup arrows, no blocked→runnable transitions, no latency chains — across 47k tokens we found literally zero. Every standard perfetto workflow for finding "who waits on whom" silently returns nothing on tron. This is the deepest reason two people can stare at the same trace and disagree: the trace does not contain the dependency information the argument is about.

Snippet 5 — check what your capture actually recorded: load matters

The same tracks, loaded vs accidentally idle

our first capture attempt ran with a mis-pointed load generator — the "serving" traces were actually idle, and the shapes differ completely

Analysis. Two operational gotchas invalidate many casual captures: (1) if the capture ran without steady load — or during warm-up, or after a request finished — the shapes mean nothing for the serving question; (2) engine threads keep a generic rinzler name until they do real streaming work, and the kernel truncates names to 15 chars (103/?-work_queue becomes 103/?-work_queu), so name-based track hunting silently misses threads. Before interpreting a trace: confirm the request rate during the capture window, and find the orchestrator by enumerating the rinzler process's threads, not by searching the name.

Snippet 6 — the evidence perfetto cannot give you: the clock A/B

The decisive experiment: change the CPU clock 48%, watch the token time

if decode were CPU-bound, per-token time would follow the CPU clock — it moved 1%

Analysis. This is how the question actually gets settled: not by reading trace shapes but by perturbing the suspected bottleneck. Cranking every CPU core 48% moved the per-token floor by 1% (and separately, capping the mesh clock — the I/O path — cost 6–7%, context-independent). If decode at this operating point were CPU-bound, those two dials would read the other way around. The CPU-sensitive part is the attention slope, which is small at 1.5k context and large at 6k — which leads to the reconciliation below.

Snippet 7 — the MEASURED FPGA lane (Jul 22) — and a correction to this page

The inference era of this page is over. tron already ships perfetto TrackEvent instrumentation on the whole FPGA job path (src/pos/driver.cpp, worker.cpp: enqueue → "Launch hardware matmul job" → tx_launch → "rx collecting results", flow-linked per request) — compiled into the production binary. Enabling it needs no patch and no rebuild: run system traced/traced_probes (tracebox works), make the sockets world-writable (rinzler runs as user positron — a root-owned 0755 socket fails silently), send the process SIGUSR1 once (registers the producer; verify with perfetto --query), then record a session with the driver category.

The correction. Earlier versions of this page inferred (snippets 2–3) that the FPGA computes through most of the iteration, because CPU real work was sparse in ~55% of the cycle and "something must be making progress." That elimination step was wrong: CPU-quiet does not equal FPGA-busy. Measured, the quiet time splits between FPGA compute and stage-handoff latency where neither side computes — DMA transfers, mailbox waits, descriptor round-trips, thread-relay lag. The bracket we measure (launch→collect) over-estimates FPGA busy, so 41% is an upper bound.

The reconciled token budget (at 98.6 tok/s/user → ~10.1 ms per iteration): FPGA-pipeline occupied ~4.2 ms (41%) + CPU attention ~2.1 ms (21%) + orchestrator serial head ~0.4 ms (4%) + ~3.4 ms (~34%) handoff/latency dead time. Cross-check: FPGA + dead time + serial ≈ 8.0 ms ≈ the 8.27 ms clock-immune floor from the length sweep — the model stands, but its floor is now decomposed: ~half FPGA compute, ~40% pure latency, ~5% serial CPU. The dead time being latency (not CPU compute) is exactly why +48% CPU clock moved nothing while the mesh/uncore cap cost 6–7%.

Revised verdict — neither CPU-bound nor FPGA-bound: pipeline-latency-bound. At the CI operating point, the largest single component of decode time is not FPGA compute and not CPU compute — it is the latency of relaying work between them, paid ~11,000 times per second per device in ~16 µs slices. The "FPGA waits a lot" observation is confirmed by measurement (~59% idle). The "therefore CPU-bound" conclusion still does not follow — the waits don't shrink with CPU clock — but the headroom implication is real and big: eliminating even half the dead time would be worth roughly +20% decode. That moves to the top of the follow-up list.

Gap interior — now measured too (ab37/ab37b, Jul 22). A driver-detail capture (2 s, loss-free, all 8 devices, 408k paired jobs) decomposed the gaps: ~30% of gap time is relay CPU work (dominated by "rx collecting results"; callback and tail-advance are small) and ~70% is covered by nothing at all — no slice on either relay thread: wire transfers, DMA, mbox-wait, dependency drains. TX-side prep contributes 0% to gaps — it is fully pipelined inside busy periods (that part of the engine is already right). Remarkably uniform: 68.8–71.3% "nothing" across all 8 devices. Two-capture honesty on occupancy: the launch→collect bracket reads 41% (ab36, heavy all-category tracing) vs 27.7% (ab37b, light tracing) — the bracket includes RX pickup lag and inflates under tracing load, so treat it as an upper bound: true FPGA compute occupancy is ≤~28%. The pipeline-latency-bound verdict only strengthens.

Worker-attention overlay (same ab36 trace, one clock) — and the instance-level bottom line. Overlaying the union of all 112 workers' attention-path slices onto the device gaps: during gap time, some worker IS computing 54% (strict work set) / 76% (incl. "Attention Ready") of the time — but that is below the workers' wall-clock baseline (68%/82%), so gaps slightly anti-align with attention rather than being caused by it; and counter-intuitively the short 16 µs gaps are the most attention-covered (~70%) while mid-size gaps (50–200 µs) are the quietest (~41–47%) — global lulls. Computing the union of everything (all 4 FPGAs + all worker attention-path activity + relay work) per instance: something computes 86–88% of wall clock; the truly global dead time — nothing executing anywhere in the instance — is 12.0–13.7% (~1.2–1.4 ms per token, conservative). The refined story: each engine individually idles a lot (every FPGA ≤~28–41% busy), the token being a dependency chain relayed across engines — so the structural win is deeper cross-layer/cross-user overlap, with the ~1.2–1.4 ms global dead time (+~14% if eliminated) as the hard floor of pure relay/wire cost.

Reconciliation. "The FPGA waits a lot for the CPU" and "the FPGA paces decode" are both true statements — about different configurations, phases, or trace-reading methods. Before comparing conclusions, pin down: model & tensor-parallel size, prompt/generated lengths, concurrency, load steadiness during the capture window, and whether "wait" was inferred from Running-state (unreliable here) or from a work/spin separation. If your colleague's captures were llama-class, tp2, prefill-heavy, or long-context, his reading and ours are the same physics.

Practical checklist for scoping tron in perfetto

  1. Record the operating point with the trace: model, tp, users, prompt/gen lengths, and the live request rate during the capture (we save a loadgen summary next to every trace).
  2. Find threads by enumeration, not name: ps -Lo tid,psr,comm -p $(pgrep -x rinzler), match loosely (work_queu), remember names appear only under streaming load.
  3. Never sum Running time — separate work from spin with perf sampling or instruction counts (snippet 1).
  4. Don't look for wakeup chains — there are none (snippet 4).
  5. Infer FPGA state from the work-phase structure, not from CPU gaps (snippets 2–3).
  6. Settle "bound-by" questions with a perturbation: core-clock A/B, mesh-clock A/B, or context-length sweep — each dial isolates one resource (snippet 6).
