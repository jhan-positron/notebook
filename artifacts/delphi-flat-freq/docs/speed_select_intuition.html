1. The two-and-a-half-gigahertz mystery

Here is the puzzle that started everything. A 144-core Xeon 6962P runs an inference benchmark. The chip is rated to turbo well past 4 GHz, power and thermals have plenty of headroom, and yet turbostat shows nearly every busy core pinned at exactly 2700 MHz — a suspiciously round number, with zero variance. Not "around 2.7" the way a power-limited core wobbles; exactly 2700, sample after sample.

The natural suspects are Intel's Speed Select features. So you check them:

$ intel-speed-select perf-profile info
    speed-select-turbo-freq: disabled
    speed-select-core-power: disabled

Both features report disabled. And still, under load, the cores sit at 2700. You disable them again for good measure. Nothing changes. This is the moment where the mental model most of us carry — "these are features; off means off" — quietly fails. The rest of this page is about replacing that model with one that actually predicts what the silicon does.

The missed intuition, up front turbo-freq and core-power are not speed features you switch off. They are managers of a table — a per-core registry of speed-limit classes called CLOS. The table is the thing that acts on your cores. Disabling the managers does not clear the table, and on this silicon the enforcement of the table's limits doesn't stop either. The only way out is to rewrite the table: re-associate the cores themselves to a different class.

2. Meet the machinery

Five mechanisms, each of which can have an opinion about how fast a core may run. You have met their names; what matters is their roles.

MechanismRole in one sentenceGranularity
PCU (power control unit)The referee on the die that actually sets each core's frequency, every millisecond.per core
HWPThe OS's request: "I'd like this core between min X and max Y." A wish, not a command.per core
SST-CP / core-power / CLOSA table of 4 classes (CLOS0–3), each with a frequency window; every core is associated with one class.per core, via class
SST-TF / turbo-freqA deal: designated high-priority cores get a guaranteed turbo grant, paid for by clipping everyone else.two groups
TRL (turbo ratio ladder)The classic rule: the more cores are active, the lower the allowed turbo for all of them.whole chip
RAPL (power limit)The wallet: the package may not exceed ~500 W sustained, whatever anyone else promised.whole package

Notice that only one of these — the PCU — actually sets frequencies. Everything else merely files paperwork with it. That observation leads directly to the first intuition.

3. Intuition #1: frequency is a minimum, not a setting

There is no register anywhere that says "core 30 shall run at 4100 MHz." Instead, each mechanism independently proposes a cap, and the PCU grants each core the minimum of all caps that apply to it. If you remember one diagram from this page, make it this one. Here is core 30 — a busy worker core on our machine, in the boot-default configuration:

Figure A — the cap gauntlet for one busy core (boot default; core 30 is in CLOS3). Every gate proposes a cap; the lowest wins.
HWP (OS request)4400OS asks for everything
CLOS window (core 30 → CLOS3)2700CLOS3 window is 800–2700 ← lowest cap wins
TF grant—core 30 isn't high-priority; no grant to offer
TRL (80 cores active)~3800active-core ladder
RAPL (package power)~4100plenty of watts left at this load
PCU grants: min(4400, 2700, ~3800, ~4100) = 2700 MHz

This explains the eerie exactness of the 2700 readings. A power limit produces wobble — 3241 here, 3260 there — because watts fluctuate. A class window produces a hard ceiling: the core slams into 2700 and stays there with standard deviation zero. You can often tell which gate won just by looking at the texture of the turbostat numbers.

Key idea Effective frequency = min over all applicable caps. Debugging a frequency problem is never "find the setting"; it is "find which gate is currently the minimum." Exact round numbers point to class windows; wobbly plateaus point to power; smooth steps by active-core count point to the TRL ladder.

4. Intuition #2: the switches and the table

Now to the heart of the matter — the intuition whose absence cost us a day. The Speed Select machinery is really two different kinds of thing, and the command-line tool does very little to distinguish them:

  • State: the CLOS table. Four class definitions (each a min–max frequency window) plus an association table mapping every one of the 288 hardware threads to a class. These are persistent registers in the package.
  • Switches: core-power enable/disable and turbo-freq enable/disable. These change how the machinery is arbitrated — but they do not erase the table.
Switches (what info shows)
turbo-freqdisabled
core-powerdisabled
The table (still there, still enforced)
CLOS0 window2700 – 4400
CLOS3 window800 – 2700
cores 0,1,18,19,36,37… → CLOS0
cores 2–17, 20–35, … (the other 256 threads) →CLOS3

The naïve model says: the switches gate the table; off means the table is ignored. The measured reality on this silicon: with both switches reading "disabled," the PCU kept enforcing the CLOS3 window. The readback and the enforcement live in different places, and only one of them was listening to the switch.

The trap, precisely "I thought disabling turbo-freq and core-power would be enough." Disabling changes the switch; the clamp comes from the table. Since nothing you disabled ever cleared the association of 256 threads to CLOS3-with-a-2700-ceiling, the gauntlet in Figure A is completely unchanged, and so is the result. The fix is to edit the table itself: 
intel-speed-select --cpu 0-287 core-power assoc --clos 0
Re-associate every thread to the roomy class, and the 2700 gate simply vanishes from the gauntlet.

If you take the state/switch distinction seriously, the whole affair becomes almost boring — of course flipping an arbitration flag doesn't delete rows from a registry. The confusion exists only because the tool presents both under one roof, and because "disabled" is such a comforting word.

5. Boot: the BIOS deals the cards

Where did that hostile table come from? Nobody on the team wrote it. The answer: the BIOS programs it at every single boot, as the implementation of a feature called Priority Core Turbo (PCT). The idea is defensible — pick the 16 best cores per package, give them a fast lane to 4400, and clip everyone else to 2700 so the fast lane is always affordable. As a default for a machine that runs 80-core inference workloads, it is catastrophic.

Figure B — the boot-default deal (one package shown, 8 representative cores; the machine has 72 per package). Under load:

Two properties of this deal matter enormously in practice. First, the 16 blessed cores per package are scattered wherever the fab found good silicon — they almost never line up with the cores your workload actually pins its threads to. Second, because the BIOS re-deals at every boot, any fix you apply is transient. Reboot the box, and the clamp is back. (This also produced a memorable red herring during debugging: a sister machine appeared "healthy" purely because someone had experimented on its table earlier and hadn't rebooted since.)

One subtlety hides in snapshot 3, and it is worth savoring: the best recipe keeps turbo-freq enabled. The first working fix disabled it and got 3900; leaving it on and merely fixing the associations got 4100 — a free 200 MHz. Why? Because SST-TF is not only a villain that clips victim cores; it is also the mechanism whose high-priority grant bypasses the TRL active-core ladder. Once every core is in the priority class, TF works for you. The feature was never the enemy. The table was.

7. Intuition #3: some switches rewrite the table

The state/switch distinction has a nasty corollary: a few operations that look like switches are actually bulk table editors, and they do not announce it. Two matter in practice:

Command that looks innocentWhat it silently does to the table
turbo-freq enable --autoRewrites the entire partition: target cores + their HT siblings → CLOS0, everything else → CLOS3, and wipes all four CLOS windows to defaults.
core-power enableResets the CLOS2 and CLOS3 window definitions. After this, the boot-default clamp cannot be reproduced just by re-associating cores — the 800–2700 window itself is gone until rewritten (or the next reboot restores it).

This is why careful tooling (the team's flat_freq_utils.sh) re-pins the CLOS windows on every apply and refuses to ever call core-power enable. When state can be mutated behind your back, you stop trusting remembered state and start asserting it. The same instinct you'd apply to a distributed system's configuration applies to a CPU's.

Rule of thumb After any Speed Select operation, don't reason about what the table should contain — read it back: core-power get-config for the windows, get-assoc for a few probe cores, then one turbostat under a pinned spin load to confirm the silicon agrees. Config readback and measured MHz are two different layers, and this whole saga began with them disagreeing.

8. Intuition #4: grants come in discrete rungs

Once the table is under control, a second-order surprise awaits: you cannot dial in an arbitrary frequency. The CLOS window is a cap, not a promise — setting a window of 2700–4400 does not get you 4400. What you actually receive is one of a few discrete grant rungs, chosen by the PCU based on who else is running. A controlled experiment made this vivid. Same 8 fast cores, same all-288-thread load, one variable changed:

Figure C — the 4400→4100 cliff. Experiment A: the other 280 threads clipped at 2700. Experiment B: identical, except the others may run to 3900.
A: fast 8 cores4400.0everyone else at 2700 → top rung granted; package at 450 W, not power-bound
A: other 280 threads2700clipped by CLOS3 window
B: the same fast 8 cores4100.0others now >2700 → grant drops a rung, despite ~50 W/pkg unused headroom
B: other 280 threads~3250lifted — and their lift demoted the fast cores
There is no 4200. The rungs on this part are 4400, 4100, the TRL ladder, and the clips. Asking for "at least 4.2 GHz" selects... 4100.

Note what experiment B rules out: the drop from 4400 to 4100 was not about power — the package had headroom to spare. It is a policy cliff baked into the TF grant logic: the top rung is only on offer while every non-priority core stays at or below the 2700 clip. The moment anyone else runs faster, the deal is renegotiated one rung down. You cannot buy the difference back with watts.

Key idea Think of frequency targets as a menu, not a dial. On this part, for realistic shapes, the menu is: 4400 (a few cores, everyone else clipped), 4100 (priority cores, ≲20 busy per 24-core power domain), ~3900 (TF off), and 2700 (the clip). Design your shape by choosing from the menu; wishes between rungs round down.

9. The final arbiter: power

Everything so far assumed the wallet stays shut. Load all 288 threads with heavy compute, and a different gate takes over the gauntlet: RAPL. The package hits its 500 W sustained limit and the PCU pays for it in megahertz — measured on this machine, a uniform slide to roughly 3.2–3.6 GHz depending on the mix, no matter how the table is written. Every clever shape converges to the same number because they are all spending the same watts.

Figure D — the gauntlet at full machine load (any table configuration).
HWP request4400still asking
CLOS window4400table fixed, roomy
TF grant4100granted, irrelevant
RAPL @ 501 W / 500 W limit~3270the wallet wins ← lowest cap
All-core heavy load ⇒ ~3.27 GHz, whoever you are.

This closes the intuition loop: the CLOS table dominates at partial load (where the 2700 clamp was robbing an 80-core workload blind), and power dominates at full load (where no table can help). The valuable regime for shaping is exactly the one inference lives in — many idle cores, a well-defined busy set.

The 3-tier result is the four intuitions in one line: the mid tier ran exactly where its window put it (busy driver cores measured 3888–3900, a perfect pin against the cap — intuition #1), it was easy to build once flat_freq_apply_tiers wrote the table honestly (#2, #3), and it bought nothing, because the only cores whose speed shows up in tokens-per-second were already on the highest reachable rung (#4). Frequency shaping is a story about the busy cores; everything else is decoration.

11. The cheat sheet

The dependency structure, compressed into one picture:

Figure F — who writes what, who reads what.
BIOS (every boot)writesthe CLOS table: PCT partition, CLOS3 = 800–2700 → the default clamp
isst assoc / configwritesthe table, one row or window at a time → the real lever
TF --auto, CP enablewrites!bulk, silent table edits → the traps
TF/CP enable-disablearbitrateswitches only — the table survives them
PCUreads alltable + TF rungs + TRL + RAPL + HWP → grants each core min(caps)

And the operational summary, as commands (this is exactly what flat_freq_utils.sh automates, with sibling handling, window re-pinning, and verification built in):

# everything fast (all 288 threads → CLOS0, TF stays on; ~4100 at partial load)
flat_freq_apply

# 2-tier: chosen cores (+HT siblings, automatic) ~4100; all else ≤2700
flat_freq_apply "27-46,51-70,75-94,99-118"

# 3-tier: fast ~4100 / mid ≤3900 (CLOS1) / rest ≤2700
flat_freq_apply_tiers "27-46,51-70,75-94,99-118" "0,24-26,48-50,72-74,96-98"

# always: verify the silicon, not the config
sudo turbostat --quiet --show Package,Core,CPU,Busy%,Bzy_MHz -i 5 -n 2

# and remember: the BIOS re-deals at every boot — re-apply after reboots
flat_freq_status ; flat_freq_revert
The four intuitions, one last time
  1. Minimum, not setting. A core's frequency is the lowest of several independent caps; find the winning gate.
  2. Table, not switches. CLOS associations and windows are persistent state; enable/disable flags don't clear them — rewriting associations is the actual fix.
  3. Some switches edit state. TF --auto and CP enable silently rewrite the table; assert state, never assume it.
  4. Rungs, not a dial. Grants are discrete (4400 / 4100 / 3900 / 2700 here); design shapes from the menu, and expect wishes in between to round down.
