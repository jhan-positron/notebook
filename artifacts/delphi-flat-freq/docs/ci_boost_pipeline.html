Unified explanation: on the OLD build, the single scheduler thread (work_queue) was PINNED to clamped rinzler cores in production (CPUAFFINITY) — capping prefill at ~790 tok/s regardless of worker clocks — while in checkerboard the same thread is UNPINNED and migrates to boosted cores, so checkerboard saw +12% and CI saw nothing. The new build (29924aa8) moved the coordinator onto fast worker cores by design, which is why our serving-path toggle NOW shows +11.8% CONFIRMED 2026-07-14: the decisive test ran — the talos harness itself (systems_test@994badfc, nightly-parity parameters, cache-cold verified via its own cached_tokens≈71) measured on the new stack: workers fast → prefill 875 tok/s (breaks the ~790 constant, +11%), decode 94.4 (≈ the boosted-era nightlies); workers clamped → prefill 736 (back at/below the historical band), decode 88.0. Worker toggle = prefill +19%, TTFT −16%, decode +7.3%, F-arms reproducible within 2%. Supporting chain as before (work_queue pinned-clamped on old build in production, unpinned in checkerboard, relocated to fast cores by PR-3070)
Bonus: two CI-harness measurement gotchas demonstrated live — (1) deterministic prompt seeds make any back-to-back rerun against a live engine a prefix-cache benchmark (observed "prefill 4745 tok/s", 647/1035 cached); (2) the KV store survives same-model reprovisioning (479 tokens still cached after an engine bounce); (3) harness workers die as opaque BrokenProcessPool on API errors CONFIRMED Direct observation, 2026-07-14 runs; SEED_OFFSET one-line patch (in our systems_test clone) is the reliable isolation — worth upstreaming, along with surfacing worker exceptions

One-sentence answer: the boost was never missing — CI's headline metric (decode @8u) lives in the one pipeline stage that is 85% waits, where +47% clock can only ever buy a few percent, and even that small real gain was cancelled the same night by a software regression; the stage where the boost is big (prefill, +12%) is either not resolved by CI's prefill≈ metric (open suspicion) or simply not what the headline reports.
