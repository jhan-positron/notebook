Tron CPU Cores and Software Threads

How config/resource-map.yaml becomes app, peer, helper, and device CPU sets, and where the Tron code launches and pins its main worker threads.

Resource Map to Runtime

1. YAML server type

Tron loads the resource map, identifies the local server by CPU vendor, physical core count, NUMA count, and optional model pattern, then sets the active server type.

physical_cores num_numa_nodes vendor model_pattern

2. Instance selects slices

--instance id,n indexes instance_slices[n - 1]. The code builds contiguous slice IDs starting at id * slices.

slice IDs = id * slices_per_instance ... + slices - 1

3. Slices choose resources

For each selected slice, C++ joins YAML entries from devices, tron_cores, dev_cores, optional helper_cores, and numa_nodes.

tron_cores -> app dev_cores -> TX helper_cores

4. CPU sets pin threads

App and device pin calls remove the lowest available CPU from their hwloc bitmap. RX and peer groups are derived by hwloc from the selected CPU, so the result follows topology order.

app_pin_thread peer_of dev_pin_thread

Thread Families

App worker pool

app_pool() is a lazy singleton. It creates app_get_num_cpus() - 1 workers and each worker uses with_app_pin_thread, which calls app_pin_thread().

Genoa96 full: 143 app cores -> 142 app workers
Genoa96 --instance 1,2: 72 app cores -> 71 app workers

Main model work

The main thread is pinned to one app CPU during startup. Per forward pass, the model enqueues main-helper and attention work onto the app pool; those are tasks, not extra permanent threads.

main + app workers = app core count

FPGA TX/RX workers

Tron keeps workers[8] for TX workers. Each active device starts one TX worker, and the TX worker starts one detached RX worker pinned to the TX core's SMT peer.

device threads = 2 x active devices, max 16 for 8 devices

TBB peer work

Before device initialization uses TBB, Tron binds the calling thread to the app-peer CPU set. TBB may create transient workers, but Tron does not expose a fixed TBB worker count in this path.

Helper cores

helper_cores are parsed into a helper CPU set and can be consumed one at a time with helper_pin_thread(). In the non-generated source searched here, this API is present and tested, but not used by a permanent production pool.

Auxiliary threads

Additional threads exist for TSC calibration, optional tracing, FUSE, scheduler queues, and token stream monitoring. They are not sized from tron_cores, and most are optional or created on demand.

How Sibling Cores Are Chosen

Not random. App and device CPU assignment is deterministic: pick_from_cpu_set() chooses hwloc_bitmap_first() and clears that bit. Sibling cores are then derived with hwloc peer_of(cpu) or find_peers(app_cpu_set).

App peers

app_set_cpu_list() parses tron_cores into app_cpu_set, then calls find_peers() to fill app_peer_cpu_set from hwloc topology.

App workers

thread_pool starts workers one by one and waits for each to start so CPU assignment stays monotonically increasing. Each worker pins itself by constructing with_app_pin_thread.

Device TX/RX pair

The TX worker gets the first remaining dev_cores CPU. It then sets dev->rx_cpu = peer_of(tx_cpu), launches the RX worker, and the RX worker calls pin_to_cpu(dev->rx_cpu).

YAML peer_cores caveat

The checked-in YAML has peer_cores, but the C++ resource map path found here does not read that key. C++ computes peers with hwloc. The shell helper config/resource-parser.sh has a separate --peer mode for tooling.
