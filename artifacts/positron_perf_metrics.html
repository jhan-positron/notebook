Positron Performance Metrics
The one-stop explainer: checkerboard, tron, nightly CI — and how industry numbers map onto ours

Written 2026-07-12. Claims that could not be pinned to a source are explicitly labeled.

TL;DR — the one distinction that prevents most metric mistakes:

Per-user tokens/s buys responsiveness: it tells you how quickly one request receives tokens while sharing the machine. Aggregate tokens/s buys capacity: it tells you how much total work the system completes per unit time across all requests. Confusing them makes either a crowded system look fast by quoting total output, or a fast single stream look scalable by multiplying its rate by user count.

Positron's checkerboard CSV columns (Parse, Generate, TTFT, TTLT) are all per-request numbers. Cerebras/Groq/Positron-Atlas marketing numbers are also per-request. MLPerf and "system throughput" numbers are aggregate. They are different currencies.

1. What "per-request" means VERIFIED

In a checkerboard run, each simulated user issues exactly one request per test config, and all users' requests are submitted simultaneously at t=0 — an open-loop single burst with no warmup and no pacing (stream_generate.hpp:351-378, 698-731, 1332-1336). Users are split evenly across engine instances, which are separate runtron processes (tp4→2 instances, tp2→4, tp1→8; run-benchmark.sh:255,336). The CSV "Users" column is the total across instances.

Each request is timed on its own timeline, producing one sample of each metric. The pipeline is: runtron logs one line per request → collect_metrics.py:39-42 scrapes them by regex → metrics.json stores {sample_count, harmonic_mean, mean} → the CSV exports the harmonic mean only (export_metadata.py:196). So:

A CSV cell = the harmonic mean, over all requests in the test, of a quantity measured on one request's own clock. That is what "per-request" means here. With 2 instances × 16 users, a cell summarizes 32 one-request samples (verified against a real metrics.json with sample_count: 8 for a 2×8u test).
Caveat (consensus round 2): the harmonic mean is applied to the latency columns (TTFT/TTLT) too, which underweights slow requests — it can conceal tail behavior and must not be read as a percentile or SLO number. The arithmetic mean sits unused in metrics.json.

Why per-request rates fall as Users rises: all N requests are in the engine at once, sharing the same forward passes and scheduler (see §2 and §4). Each request's own timeline stretches, so its personal tokens/s drops — even while the machine's total output climbs.

2. The life of one request: parsing, prefill, generate VERIFIED

Terminology, straight from the code: "parse" is Positron's word for prefill — they are exactly the same thing. "Generate" is a timing window inside decode. The full lifecycle:

  1. Tokenize (CPU, before submission). Prompts are encoded to token IDs up front — before the request clock starts (stream_generate.hpp:1103-1117; rinzler tokenizes in-process, including chat-template application: rinzler.cpp:2088-2094). Checkerboard's TTFT therefore excludes tokenizer time.
  2. Prefill == parse. The full submitted prompt — shared system prompt S plus user prompt P (verified: p2048+s512 logs "Request with 2560 tokens in") — is pushed through the model in chunks of 128 tokens per user (libtron.hpp:197-205) to build the KV state. When the prompt is fully processed, on_start fires (libtron.hpp:294) and the final prefill logits are sampled: the first output token comes "for free" from parsing (stream_generate.hpp:231-232).
  3. Decode == generation. One token per engine forward pass per request, continuously batched with everyone else's work (§3), until exactly G tokens exist (checkerboard forces --dont-stop -l G; run-benchmark.sh:265).

What each phase does, and on which silicon

From the tron design document (doc/tron_design_document.md:60-80) and code: the FPGAs run all matrix multiplies (WQ/WK/WV, WO, the MLP W1/W2/W3, and the logits projection Wcls). The host CPU runs everything between the matmuls: embedding lookup, RMSNorm, RoPE, residual adds, attention, and sampling. Hardware attention exists but is off by default (USE_HW_ATTN, hw_attn_config.hpp:13-22) — and even when enabled, prefill attention must run on the CPU (multi-query edges are software-only, full.hpp:1493-1500; decode attention is HW-eligible only past token 128).

Why this matters: prefill attention cost grows with context length and is unconditionally CPU work executed by the app-pool "attention worker" threads on the tron cores. This is the mechanism behind our measured frequency results: boosting worker cores 2700→~4100 MHz lifted parse rates by up to +43% on long-prompt/high-user configs (near the 4100/2700−1 = 52% clock-ratio ceiling) but only ~+12% on short/light configs — the uplift map is a CPU-boundedness map.

3. Checkerboard (tron-direct) vs production (rinzler) VERIFIED

A common misconception (we held it too): rinzler is not a proxy in front of runtron. rinzler is the engine process — it links the same libtron/alderaan inference library and drives the FPGAs in-process, adding an HTTP frontend (README.code-org.md:16-17; rinzler.cpp:1706-1719, 3173-3218). runtron is the dev-CLI frontend over the same library. So checkerboard and production run the same engine, scheduler, batcher, and FPGA path; what differs is everything around it:

Checkerboard (runtron direct)Nightly CI / production (rinzler)
Engine, scheduler, batching, FPGA pathidentical (same libtron library)
Request entryin-process call, pre-tokenizedHTTP + chat template + tokenize on the clock
Load shapeone burst; each user exactly 1 requestreal arrivals; CI: sustained closed-loop-style load @8 users
What TTFT includesinternal queueing + prefill only+ network, LB, HTTP, SSE, admission effects
Overload behavioreverything queues internallyadmission rejection (429), not queueing
Reported metricsParse / Generate / TTFT / TTLT (per-request HM)per-user decode TPS @8u; "prefill" ≈ prompt_len / client TTFT
Sees rinzler's work_queue placement?n/a (thread lives in runtron itself)yes — scheduler thread runs on the pinned rinzler cores

The hot thread production observers see at 87–100% CPU on rinzler cores — named like 72/?-work_queue (work_queue.hpp:446-448) — is the scheduler's single worker per model: it drains requests, mutates the shared token tree, and launches every forward pass back-to-back (work_queue.hpp:316-360). Unlike the TX/RX DMA threads (busy-looking mwaitx poll loops), this occupancy is real scheduling work under load.

4. System aggregate throughput — and why you can't get it from the CSV DERIVED / PROXY

Aggregate throughput answers "how many tokens does the whole box produce per second across all users?" Checkerboard does not report it, and it cannot be recovered from the CSV: the CSV stores harmonic means of per-request rates, and true aggregate needs the burst makespan (first submission → last completion), which isn't exported. (The engine can log per-pass throughput under TRON_PERF_INTERVAL_SEC, full.hpp:205-227, but checkerboard doesn't collect it.)

Reading it against the workload knobs

  • Users ↑: per-request rate falls (requests share forward passes and queue behind each other), aggregate rises toward saturation. Neither number alone describes the system — quote both.
  • User prompt p ↑: per-request parse rate has a hump (amortize fixed costs, then attention's superlinear context cost bites — visible in both clamped and boosted runs at p512).
  • Shared system prompt s: submitted with every request and always counted in Parse's numerator — but the engine's prefix tree may serve those tokens from KV cache (token_cache.hpp:30-90, tokens_reused counters), so Parse is a logical input-token rate that can exceed tokens physically computed. VERIFIED mechanism; exact simultaneous-parse semantics INFERRED.
  • Grid confound: in our standard sweeps s is zipped to p/4 and g=p — so s, p, and g effects are confounded by design. Treat rows as a workload family, not as independent-variable scans.

5. How industry numbers map onto ours VERIFIED (references below)

The industry has converged on a small vocabulary, defined most cleanly by NVIDIA's benchmarking guide[2] and operationalized by Artificial Analysis (AA), the de facto scoreboard[6]:

Industry metricDefinitionNearest Positron metricMapping caveats
Output Speed (AA); "tokens/s per user" (Cerebras[7], Groq[8], Positron Atlas[11]); TPS/user (NVIDIA) Per-request output tokens/s after the first token; AA reports median (P50) over network Generate (tok/s) — per-request decode-window rate; nightly CI's "decode TPS @8u" Ours: harmonic mean, in-process, excludes first and ~last token. Theirs: median, over network+SSE. Must match concurrency to compare — AA is near-single-stream; our CSVs span 2–32 users.
TTFT (AA, NVIDIA, vLLM[4]) Request sent → first token received TTFT Checkerboard TTFT excludes tokenizer, HTTP, network, LB, SSE (in-process clock). Nightly CI TTFT includes them.
ITL / TPOT (e2e − TTFT)/(output tokens − 1); per-user speed → 1/ITL ≈ 1 / Generate Same window philosophy (excludes first token).
Prefill throughput (no canonical standard; e.g. Anthropic's 21K tok/s input claim[9]) Prompt tokens processed per second Parse (tok/s); nightly CI's prompt_len/TTFT Ours is queue-inclusive ((S+P)/TTFT), counts cached shared-prefix tokens (logical rate), in-process. Three caveats — always state them.
System throughput (NVIDIA aggregate TPS; MLPerf offline[10]) Total tokens / wall-clock window across all requests None exported. users×HM is only a proxy (§4) Do not present N×HM as aggregate.
MLPerf server scenario[10] Max tokens/s under Poisson arrivals with TTFT ≤ 2s AND TPOT ≤ 200ms SLOs Nightly CI goals (decode TPS floor per user @8u) Analogous SLO framing — not equivalent (different arrival process, SLO set).

Who quotes what

  • Accelerator vendors lead with per-user speed: Cerebras "1,800 tokens/s on Llama 3.1-8B", "2,522 TPS/user on Llama 4 Maverick vs 1,038 for NVIDIA Blackwell"[7]; Groq "300 tokens/s per user on Llama-2 70B", AA-verified 276 t/s on Llama 3.3-70B[8]. These are all the AA Output-Speed family — the same family as our per-request Generate.
  • Model companies mostly quote relative speed ("GPT-4o is 2× faster", "Haiku 4.5 is 4–5× faster than Sonnet 4.5") and leave absolute t/s to third parties; DeepSeek-V3's "60 tokens/second" is the notable absolute exception[9].
  • Positron's public Atlas benchmark uses exactly the per-user family: "280 tokens/sec/user on Llama 3.1-8B (BF16, no speculation) vs 182 tokens/sec/user for NVIDIA DGX H200", plus system framing: 4.54× perf/W and 3.08× perf/$[11]. Relative to internal metrics: that number lives in the Generate/decode-speed family, measured at the concurrency stated in its methodology (not asserted single-stream here), with perf/W an MLPerf-server-like system dimension our CSV doesn't carry. It is not a Parse/prefill number and not an aggregate number.
The three-sentence rule (consensus, verbatim): Per-user tokens/s buys responsiveness: it tells you how quickly one request receives tokens while sharing the machine. Aggregate tokens/s buys capacity: it tells you how much total work the system completes per unit time across all requests. Confusing them makes either a crowded system look fast by quoting total output, or a fast single stream look scalable by multiplying its rate by user count.
