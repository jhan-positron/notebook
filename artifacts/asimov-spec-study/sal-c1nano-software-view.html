Answers at a glance

QuestionShort answer
How does C1-Nano software send work to SAL? Ordinary stores (or DMA bursts) into a 20-partition, 80 KB SPA write window. The SAL Packet Assembler converts those AXI4-MM writes into 256-bit AXI-Stream packets that SAL's command interface consumes. See Q1–Q2.
What commands exist? Eight tdest types (RTL enum): NOOP, SAL_CMD, CPS_METADATA_CMD, CPS_ACTIV_CMD, CPS_WEIGHTS_CMD, CPS_ROUT_CMD (a bare 176-bit rd_t), SAL_LO_PRI_CMD, RSVD. Only CPS_WEIGHTS_CMD bypasses the queues. See Q1.
What is SPA and why does it exist? An impedance-matcher and multiplexer: many software agents (C1 threads, DMA channels) → one ordered packet stream, with per-partition isolation, template reuse, and MMU-enforceable ownership. See Q2.
What are the packet formats? SAL commands: 4 × 128-bit persistent slots = 512 b per packet, slot 3 is the trigger. CPS commands: 256 b in two half-slots, four types. In-band per-lane tags select type/slot/half. SAL_CMD is now fully bit-defined (489-bit sal_cmd_t, sal_cmd.h + uArch §5.2.6.2); CPS payloads remain opaque to SAL. See Q3.
How does software pace and observe SAL? FSR counters: HP/LP queue depth, SPM read/write, AMem write, plus SPA per-type progress pulses. High-priority queue strictly preempts low. See Q4.
What else feeds SAL besides software? SPM RG records (send_to_sal egress queue, 1-credit interface) and SPM vector data (2 kb read/write interfaces). SVL does not feed SAL directly. See Q5.

Q1

The command path: stores in, packets out

Software never talks AXI-Stream, and SAL never parses loads/stores. A C1-Nano thread (or a DMA descriptor it programs) simply writes into its SPA partition — a 4 KB window of the address map. The CMN carries those writes over an HN-I port (AXI4-MM, 256-bit) to the SAL Packet Assembler, which assembles them into well-formed 256-bit AXI-Stream packets and hands them to SAL's command interface in order.

The RTL command-type enum (cmn_sal_cmd_type_pkg, carried as AXIS tdest) refines the docx's three families:

  • SAL_CMD / SAL_LO_PRI_CMD — two-beat 512 b commands for SAL itself. The "load AMem" work is inside the SAL command's exec descriptor (exec.amem_wb + exec.amem_addr), not a separate command.
  • CPS_ROUT_CMD — a single-beat command whose payload is a bare 176-bit rd_t routing-data record; CPS_METADATA_CMD carries a 226-bit attention-metadata payload. Both ride the high-priority queue as opaque beats and are forwarded to CPS.
  • CPS_ACTIV_CMD — "process this batch". Since May 2026 it rides the HP queue too (the old direct passthrough is gone); cps_cmd_adapter additionally delays its release to CPS until the preceding AMem writes are architecturally visible (propagation delay + CSR-tunable cps_activ_cmd_delay) — this ordering guarantee is the reason the command path routes through SAL.
  • CPS_WEIGHTS_CMD — the only type that bypasses both queues (dedicated skid path), merged at the pipe tail with priority weights > released ACTIV > other CPS commands.
Q2

SPA mechanics: partitions, slots, triggers

  • 20 partitions, one owner each. Each partition is owned by one software agent — a C1 thread or a DMA channel. Ownership is a software convention, but the 4 KB partition size deliberately matches one MMU page, so the OS can make it enforceable. Inside a partition the address offset is ignored: per-lane in-band fields in the written data select the command type, slot, and half.
  • Template stashing. Each partition holds 4 persistent 128-bit SAL_CMD slots (a 512-bit packet) plus an independent set of 4 for SAL_LO_PRI. Slots keep their value until overwritten, so software pre-loads a command template once and rewrites only the fields that change per issue — e.g. write slots 0–3 for packet 0, then only slot 3 for packet 1.
  • Slot 3 is the trigger. Writing it fires the packet: the trigger write lands, then a snapshot samples the full 4-slot bank — including the trigger itself — and locks it against later body writes. On emission, slots pack two per 256-bit beat (0+1, then 2+3) with in-band bits left in place.
  • CPS commands are 256-bit payloads in lower/upper half-slots, one pair per (partition, type) for the four types CPS_METADATA, CPS_ACTIV, CPS_WEIGHTS, CPS_ROUT. The lower half persists across commands. Egress packs beat[123:0] = half 0, beat[247:124] = half 1, beat[255:248] = 0 — the in-band tags never reach the wire (shadow registers still read back raw lanes, tags included).
  • DMA-friendly. A single burst may mix command types (each 128-bit lane carries its own cmd_type), AXI preserves beat order within a burst, and software must simply order the trigger beat last. The SPA does not roll back clean beats committed before a later burst error.

Queue geometry (RTL): HP queue 4096 entries, LP queue 1024 entries, both of 512-bit command entries; overflow pushes are silently dropped with a sticky IRQ. A full invalid-command capture unit records framing/reserved-type/LP-exec violations with cause codes.

Q4

Pacing and observability

SAL runs two command queues — HP 4096-deep (SAL_CMD + queued CPS types), LP 1024-deep (SAL_LO_PRI_CMD only, restricted to write-back work) — with strict priority among ungated heads: an HP head waiting on its 19-bit prog_cnt dependency releases the LP queue to overtake it (LP's own gate is CSR-switchable, default on). The functional labels (RMS norm vs all-reduce etc.) are software conventions; the RTL opmodes are GEN/NORM/ACCUM/ACCUM_ADD_LATENT/PE/LPA/PW/TOP96 on either queue. Software paces itself through FSR read over each DSU's Peripheral Port (64-bit AXI5, 1 MB aperture):

  • FSR holds 17 event counters + 4 live queue-depth mirrors (16-bit, cmn_fsr_main.rdl): SAL HP/LP × {SPM read, SPM write, AMem write}, 4 CPS completion counters, the SVL results counter, and 6 SPA per-packet-type counters.
  • The wait idiom is preset-and-poll-for-zero: software writes 0x10000 − K into a counter and polls until it wraps to 0 after K events.
  • SPA's pulses fire on completed packet egress (AXIS tlast handshake toward SAL) — ingestion is observable separately from execution.
  • The SPM read/write events are counted outside SAL (SAL tags each SPM request with its queue of origin); 32-bit debug views of the same counts live in the sub-block CSRs.
Q5

What reaches SAL besides the software command path

  • SPM RG records — SAL owns one of SPM's seven egress queues (send_to_sal, credit bank of 16 = SPM_RG_INIT_CREDIT). RTL detail (wait_mdata.sv): incoming md_t records are demuxed into four 256-deep FIFO pairs by their spm_source field (SVL/SAL/C2C/CMN); SAL data-load commands then consume them — comparing the record's id.hw_id against the command's expected ID, and (with use_mdata_addr) taking the SPM address from the record's spm_addr (or c2c_spm_rwrite_addr for C2C-sourced records) and the transfer count from datagram_size. A save_mdata/mwb_* mechanism lets a command re-emit the record with overwritten hw_id/send_to fields.
  • SPM vector data — SAL reads latents and partial tensors over its 2 kb read interface and writes results back over its 2 kb write interface (the high/low-priority SPM counters above track exactly this traffic).
  • Not SVL. SVL and SAL are siblings in SLS but do not feed each other directly — SVL's outputs reach SAL only by way of SPM (vectors + RG records) or AMem/CPS orchestration.
Q6

Registers — the docx address map is superseded

RTL correction: the generated HRA map (hra_addr_map_pkg.sv, 30-bit aperture-relative addresses) gives SAL a 4 MB window at 0x2040_0000–0x207F_FFFF — not 4 KB at 0x236000. Neighbors: SVL 4 MB @ 0x2080_0000, SPM 4 KB @ 0x2003_1000, SSA 16 KB @ 0x2003_4000, CPS @ 0x2003_0000, C2C @ 0x2000_0000. The 4 MB is real, not padding: SAL's CSR map embeds external SRAM programming windows — theta_lut @ +0x100000 (131072 entries, RoPE angles), LPA L1/L2 LUTs @ +0x200000/+0x210000, sal_cfg @ +0x220000 (1024 circuit configs selected per-command by exec.conf_idx), and two 32-entry clamp-descriptor SRAMs. Notable core CSRs: ctrl (incl. en_lo_pri_q_cnt_gate), out_fp_fmt (must match the SPM consumer), cps_activ_cmd_delay, queue-depth/AMem/SPM counters, FIFO status, two-group IRQ block, invalid-command capture, and a telemetry adapter with 74 slots. Inside the SLS CSR daisy-chain the leaves sit at SSA 0x00_0000 / SAL 0x80_0000 / SVL 0xC0_0000.

Resolved (was an open item): the SPA AXI-Stream is the SAL command path, full stop. The AXI-Lite bridge that CMN spec §4.4 described does exist in RTL — but as cmn_hra_axi_axil_adapter, the HN-I→HRA register-access chain, not a command path. §4.4's "root point for all commands" wording is stale and should be rewritten to point at the SPA. Note also the docx's summarized 0x233000–0x237000 table matches nothing in the generated map (the SPM note's 0x234000 claim inherits this staleness — the RTL SPM window is 0x2003_1000).

MechanismTransparent?What software sees
AXI4-MM → AXI-Stream conversion, packet framing, no-interleave guaranteeYesNothing — write slots, packets come out well-formed.
Snapshot capture on triggerYesGuarantees the fired packet excludes later body writes; no action needed beyond trigger-last ordering.
In-band lane tags (cmd_type, slot, half)NoSoftware crafts them in every written lane; they are consumed by SPA and dropped at egress.
Slot persistence / templatesNoA software feature to exploit: rewrite only changed slots, trigger last.
Partition ownershipNoConvention + MMU mapping; HW does not enforce single-writer.
