SVL from the C1-Nano software perspective: configure it, observe it, consume its output — never command it

Answers at a glance

QuestionShort answer
Does C1-Nano interact with SVL? Yes — but only indirectly. There is no command queue into SVL and no data path from a core to it. Software touches SVL on exactly three surfaces: CSRs, FSR counters, and its generated data, delivered via SPM/ACP. See Q1.
Who drives SVL, then? The TWE. SVL is a stream processor bolted to the systolic array's output: dot-product vectors arrive with Routing Data traveling in sync (injected by software much earlier, via SAL→CPS command words). That routing data is software's only per-vector influence on SVL. See Q2.
What can software configure? SVL CSRs behind a 4 MB HRA window at 0x2080_0000 (docx's 0x237000/4 KB is stale). The CSR list is published in svl_csr.rdl: top-k depth (7-bit, reset 8), output format, LPA descriptors + L1/L2 LUTs, 32 clamp descriptors, 128 softcap entries, IRQ/telemetry blocks. See Q3.
What does software get back? Per-vector Generated Data — topk_gd_t (8 × {8 b idx, 16 b val}) or attn_gd_t (19 b Σz, max), bit-defined in mdata_pkg.sv — pushed to a chosen core through SPM's ACP path, plus the FSR SVL Results Counter. See Q4.
Attention vs non-attention? Refined by RTL: during the attention K-pass SVL writes only to EMem — nothing reaches SPM. But the V-pass (ATTN_SM_V) does write results to SPM with an md_t carrying Σz/max generated data. Outside attention every vector lands in SPM with an md_t. See Q5.

Q1

Is there any direct C1-Nano ↔ SVL interaction?

No direct data or command interface exists. SVL has no command queue, no SPA-like front door, and no ACP/interrupt channel of its own. It is a stream processor hard-wired to the TWE's accumulator outputs: 512 columns of 19-bit dot products arrive every clock, get processed, and leave — toward EMem during attention, toward SPM otherwise. Software cannot address SVL as a bus target except for its CSR block, and SVL never initiates traffic toward a core except through the SPM metadata machinery.

The interaction that does exist is three-sided: configure (CSRs), observe (FSR), consume (generated data). Each is one small surface, detailed below.

Q2

Software's real control knob: routing data injected upstream

The RTL makes the "two-channel" split explicit (sls_svl_uarch.md §5.1.5): the CSR plane holds retained configuration (table contents, top-k depth, output format), while everything per-vector arrives as the CPS passthrough context riding the TWS corner sideband in sync with the accumulator data (cps_sls_pkg::svl_passthru_t): a 4-bit svl_opmode per beat (SVL never latches a command), clamp enable + descriptor index (1 of 32), LPA descriptor index (1 of 64), softcap index (1 of 128), the K-token window + 64-bit speculation mask, and the 176-bit routing data. So table contents are CSR-programmed; table selection is per-command. SVL forwards the routing data essentially untouched — the RTL overrides exactly three fields: spm_source = SPM_SRC_SVL, gd_valid (set only for TOPK and ATTN_SM_V), and datagram_size = 256. The control loop remains long: C1-Nano → SPA → SAL → CPS → (rides with TWE outputs) → SVL → SPM → ACP → C1-Nano — per-vector steering, always one command-batch ahead, never interactive.

Q3

Surface 1 — CSR configuration (now fully published)

Path: C1-Nano → HRA register chain → SVL CSR block — a 4 MB window at 0x2080_0000 per the generated hra_addr_map_pkg.sv (the docx's 0x237000/4 KB matches nothing in the RTL map; within the SLS CSR daisy-chain SVL is the leaf at 0xC0_0000). The register list from svl_csr.rdl:

  • ctrl @0x00 (soft_reset, stat snapshot) · build_version · scratch · out_fp_fmt @0x18 (BF16/FP16 for SPM output — must match the consumer) · topk @0x20: 7-bit, reset 8, valid 1–8 (0 clamps to 1, >8 raises topk_cfg_err when a TOPK op runs) · svl_result_cnt @0x28 (32-bit RO).
  • Two-group IRQ block (norm + critical; critical-only: invalid_opmode, fifo_protocol_err, topk_cfg_err) · SRAM trim/test · telemetry adapter (58 slots + 32 matchers).
  • Math-table windows (all write-only; reads return a 0xDEADBEEF sentinel — no readback verification): lpa_desc @0x1000 (64 descriptors × 4 words, word-3 write commits), 32 clamp_desc @0x6000, 128 softcap entries @0x7000 (c1/c2 for sc = c1·tanh(c2·x)), LPA L1/L2 LUTs @0x200000/0x210000 (1024 × 64 b each; L2 = three BF19 Horner coefficients).

Resolved (was a gap) — with new software obligations. The CSR set is fully enumerated in svl_csr.rdl + sls_svl_uarch.md. But note: SVL hardware enforces only two field-level checks (invalid opmode, top-k depth); every other programming rule is a software contract whose violation produces silently wrong data. The write-only table windows cannot be verified by readback, the LPA staging register carries stale bytes into partially-written descriptors, and reprogramming topk mid-command is a sequencing violation.

Q4

Surfaces 2 & 3 — observe via FSR, consume via ACP

  • FSR — the SVL Results Counter increments once per 128-element vector written from SVL to SPM, timed to SRAM-write completion. It lives as a 16-bit FSR counter (lane 3 of cmn_fsr_main.rdl, wait idiom: preset 0x10000−K, poll for wrap-to-zero) and is mirrored into the SVL CSR svl_result_cnt @0x28 (32-bit; RTL nuance: the count is SAL-sourced at the SPM boundary and mirrored with one register of latency). Read over the DSU Peripheral Port like all FSR state.
  • Generated Data — for each non-attention vector, SVL computes descriptives so software doesn't have to: topk_gd_t (8 lanes of {8-bit index, 16-bit value} — MoE router gates, final logits) or attn_gd_t (Σz sume and smax, 19 b each). These ride in the md_t record (bits [367:176]) and reach the selected core as the second 32 B ACP write — already stashed in that core's L2 by the time software looks. Bit layouts are pinned in mdata_pkg.sv; validity is gated by rd.s.gd_valid.

Net effect: the top-k/sum/max work that would otherwise be a software reduction over every 256 B vector arrives precomputed, cache-warm, and addressed to the right core — that is the entire point of the SVL↔software relationship.

Q5

Attention vs non-attention: when software sees nothing

Mode (RTL opmode)SVL outputSoftware visibility
Attention K-pass (ATTN_SM_K / _SC / _GELU)exp(x−max) → EMem (BF19); Σz/max stats → TWSNone per-vector — the K-pass opmodes push no routing data, no metadata, no SPM traffic (RTL suppresses the whole sideband path).
Attention V-pass (ATTN_SM_V)results → SPM + md_t with attn_gd_t (Σz, max)Correction vs docx: the V-pass does write SPM and delivers generated data — software sees it like any non-attention vector.
Non-attention (SIGMOID/SILU/EXP/RECIP/TANH/GELU/TOPK/LPA/INDEXER/BYPASS)128-elem vectors → SPM + md_tFull: Results Counter, RG record via ACP (if send_to_cmn), vector readable in SPM. Only TOPK sets gd_valid with topk_gd_t.

Notes from the RTL: only TANH/GELU/LPA are LUT-based — SIGMOID and SILU are composed from the EXP/ADD/RECIP(×MULT) primitives. The full opmode enum has 14 modes including BYPASS, LPA (programmable piecewise polynomial), and INDEXER (ReLU→weight→accumulate, with its weight SRAM loaded through a SAL backdoor). SVL's outputs go to SSA, which assembles the md_t and owns the column-condenser (the CCF is in SSA, not SVL) — and the SVL→SSA/SPM path truly has no backpressure: SSA ties SVL's credit to constant 1, and a top-k generated-data entry that collides with a V-pass push is silently dropped (telemetry pulse only).
