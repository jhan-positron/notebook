SVL from the C1-Nano software perspective: configure it, observe it, consume its output — never command it

Compiled 2026-07-09 from Asimov_Arch_Spec (§5.1.4 SVL, §5.2.6 specialized acceleration, §5.4.3 inference flow, §5.7, §5.9 progress counters), CMN_uArch_Spec.docx (§11 FSR, §4.8 ACP), SPM spec §5.5, and mdata_pkg.sv (refmod_exp). Companion to the SPM and SAL notes; same conventions.

Answers at a glance

QuestionShort answer
Does C1-Nano interact with SVL? Yes — but only indirectly. There is no command queue into SVL and no data path from a core to it. Software touches SVL on exactly three surfaces: CSRs, FSR counters, and its generated data, delivered via SPM/ACP. See Q1.
Who drives SVL, then? The TWE. SVL is a stream processor bolted to the systolic array's output: dot-product vectors arrive with Routing Data traveling in sync (injected by software much earlier, via SAL→CPS command words). That routing data is software's only per-vector influence on SVL. See Q2.
What can software configure? SVL CSRs behind the HRA window at 0x237000 — e.g. top-k extraction (k ≤ 8, a 7-bit CSR field), LUT/LPA-based function setup. The full CSR list is not yet published. See Q3.
What does software get back? Per-vector Generated Data — topk_gd_t (8 × {8 b idx, 16 b val}) or attn_gd_t (19 b Σz, max), bit-defined in mdata_pkg.sv — pushed to a chosen core through SPM's ACP path, plus the FSR SVL Results Counter. See Q4.
Attention vs non-attention? During attention (QKT) SVL writes only to EMem — nothing reaches SPM, so software sees nothing per-vector. Outside attention every 128-element vector lands in SPM with an md_t record. See Q5.

Q1

Is there any direct C1-Nano ↔ SVL interaction?

No direct data or command interface exists. SVL has no command queue, no SPA-like front door, and no ACP/interrupt channel of its own. It is a stream processor hard-wired to the TWE's accumulator outputs: 512 columns of 19-bit dot products arrive every clock, get processed, and leave — toward EMem during attention, toward SPM otherwise. Software cannot address SVL as a bus target except for its CSR block, and SVL never initiates traffic toward a core except through the SPM metadata machinery.

The interaction that does exist is three-sided: configure (CSRs), observe (FSR), consume (generated data). Each is one small surface, detailed below.

Q2

Software's real control knob: routing data injected upstream

SVL decides what to do with each vector based on the Routing Data that travels in sync with it — the same rd_t from mdata_pkg.sv. That routing data enters the machine when software posts an Activation Cmd through the SPA→SAL→CPS path (see the SAL note): the CW "includes routing data information to be conveyed in sync with TWE output vectors so downstream modules know what each vector is (model, matmul, user …) and where to store it in SPM" (Arch spec §5.1.4). So the control loop is real but long: C1-Nano → SPA → SAL → CPS → (rides with TWE outputs) → SVL → SPM → ACP → C1-Nano. Software steers SVL per-vector, but always one command-batch ahead, never interactively.

Q3

Surface 1 — CSR configuration

Path: C1-Nano → CMN SBSX (ACE5→AXI4-Lite) → SVL CSR block, HRA window 0x237000–0x237FFF (beside SAL at 0x236000). Documented knobs:

  • Top-k extraction: k ≤ 8, "per CSR 7-bit field" (Arch spec §5.2.6) — sets how many top elements SVL reports per vector into Generated Data.
  • LUT-based functions (TANH, GELU) and the LPA (LUT-based Polynomial Approximation) module — table contents/coefficients are software-loadable by design intent.

Gap: no spec yet enumerates the SVL CSR set — the register list, which functions are CSR-selected versus routing-data-selected, and the LPA table load mechanism are all unpublished. The 7-bit top-k field is the only concretely documented CSR today.

Q4

Surfaces 2 & 3 — observe via FSR, consume via ACP

  • FSR — the SVL Results Counter increments once per 128-element vector written from SVL to SPM, timed to SRAM-write completion (Arch spec §5.9.2). Read over the DSU Peripheral Port like all FSR state. This is software's pacing signal for "how many partial tensors have landed."
  • Generated Data — for each non-attention vector, SVL computes descriptives so software doesn't have to: topk_gd_t (8 lanes of {8-bit index, 16-bit value} — MoE router gates, final logits) or attn_gd_t (Σz sume and smax, 19 b each). These ride in the md_t record (bits [367:176]) and reach the selected core as the second 32 B ACP write — already stashed in that core's L2 by the time software looks. Bit layouts are pinned in mdata_pkg.sv; validity is gated by rd.s.gd_valid.

Net effect: the top-k/sum/max work that would otherwise be a software reduction over every 256 B vector arrives precomputed, cache-warm, and addressed to the right core — that is the entire point of the SVL↔software relationship.

Q5

Attention vs non-attention: when software sees nothing

ModeSVL outputSoftware visibility
Attention (QKT = S)E’s → EMem (BF19)None per-vector — "during QKT=S, SVL does not write anything to SPM". Only downstream counters move.
Non-attention128-elem vectors → SPM + md_tFull: Results Counter, RG record via ACP (if send_to_cmn), vector readable in SPM.

The attention loop (TWE → SVL → EMem → TWE) is deliberately closed in hardware; software orchestrates it only at command granularity and observes it only through CPS/TWS progress counters.
