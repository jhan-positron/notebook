Q1

How software on C1-Nano accesses SPM memory and metadata

Path 1 Data memory (16 MiB) — memory-mapped, direct load/store or DMA

The 16 MiB vector memory is a slave on the CMN fabric: C1-Nano → DSU (RN-F) → CMN mesh → four AXI4 slave ports (256-bit data bus each, exposed by CMN SBSX nodes, listed in the SPM spec as “(8) 256-bit AXI full duplex interfaces”) → spm_cmn_if in spm_south, which converts AXI transactions to SPM’s internal credit-based protocol. Aggregate bandwidth is 256 GB/s. Software can either issue plain loads/stores to the SPM window or program one of the 4 CMN DMA engines dedicated to SPM↔CMN transfers.

  • Writes: AXI beats of 128/256 bits only; any aligned 16 B or 32 B chunk of a 256 B word can be written (the adapter accumulates up to eight beats per port into a 2 kb chunk before dispatch).
  • Reads: AXI read requests are split into 2 kb reads; the array always reads a full 2048-bit word internally and returns the requested slice. So a 16 B read costs a full-word access — batch reads when you can.

Path 2 Metadata (RG data) — pushed to the core, not fetched by it

In normal operation software never reads metadata out of SPM. When a producer writes a vector, its 46-byte RG record lands in the RG memory and, if send_to_cmn is set, the entry’s address is queued in one of five per-DSU virtual queues (acp_dsu_sel picks the DSU). When credits allow, SPM reads the entry back and delivers it through the CMN to the target DSU’s Accelerator Coherency Port (ACP) — a “backdoor” write directly into the core’s coherent cache, once per vector. Arrival is also observable by polling the FSR progress counters (each module publishes raw 16-bit counter values to the CMN FSR module; SPM’s write counter increments on SRAM-write completion).

  • Writing metadata from software: possible — CMN is a first-class RG producer. A metadata-only write through the CMN AXI port carries the target entry address in its spm_addr routing-data field. This is how ARM cores inject work notifications toward SAL or C2C.
  • Reading metadata memory: debug only, via AXI-Lite indirect registers — write address + start bit to rg_data_fifo_ctrl, poll rg_data_fifo_done, read the entry from rg_data_fifo_data[6] (6×64-bit = the 368-bit entry). Virtual-queue contents are readable the same way via rg_data_vfifo, and queue read/write pointers via vfifo_ptr[7].

Path 3 Registers — AXI4-Lite via the CSR network

Covered in Q5 below.

Q2

Is “SPM metadata” the same thing as routing data?

Almost — routing data is the mandatory core of it. The SPM uArch spec’s precise term is RG data: “routing and generated data”. One 46 B RG record accompanies every 256 B vector:

  • Routing Data — identifies the vector (model, matmul, user, TP group/ID), says where it lives (spm_addr), and where its record must go next (send_to_sal / send_to_c2c / send_to_cmn + acp_dsu_sel, C2C Tx flag). This is what the delivery hardware acts on.
  • Generated Data — optional per-vector descriptives computed by SVL to offload software: top-k (k ≤ 8), sum (Σz), max. Example from the arch spec (DeepSeek V4): a 24-byte record = top-8 values (8×2 B) + top-8 indices (8×1 B), used for the MoE router gate and final logits.

So when the SPM spec says “meta-data (routing data)” it is using the terms loosely-interchangeably; strictly, metadata ⊃ routing data. In conversation with the SPM team, “RG data” is the unambiguous word.

Q3

Is SPM metadata memory-mapped to C1-Nano?

No. The RG memory does not appear as a directly addressable load/store region in any address map. Access is asymmetric by design:

DirectionMechanismMemory-mapped?
HW → software (consume)ACP push of the 46 B record into the selected core’s cache; FSR counters signal arrivalNo — pushed
Software → HW (produce)Metadata-only write through the CMN AXI data ports; entry address rides in the record’s spm_addr fieldVia data ports
Software → RG memory (inspect)AXI-Lite indirect: rg_data_fifo_ctrl/rg_data_fifo_data[6], rg_data_vfifo, vfifo_ptr[7]No — indirect, debug

The rationale: metadata is a work-notification stream, not a data structure to poll. The virtual queues + credit-gated delivery give ordered, loss-free hand-off without software ever taking a cache miss on SPM SRAM.

Q4

Who accesses SPM from CMN? (the page-3 diagram)

Your reading of the diagram is correct: the “AXI to mdata/2Kb bus” module (implemented as spm_cmn_if in spm_south) accepts CMN AXI transactions and steers them to both the 16 MB data memory (via the 2 kb bus / memory manager) and the metadata engine (spm_mdata). There is no contradiction with Q3 — the distinction is which transactions each requester can usefully issue:

  • Requesters behind those four AXI ports — anything mastering the CMN mesh: the 10 C1-Nano cores (5 DSUs, RN-F nodes), the 4 SPM↔CMN DMA engines (behind RN-I nodes), the host CPU (PCIe Gen6 → RN-I), and at boot the Cortex-M7 SCP.
  • To the data memory: normal reads and writes (16/32 B write granularity) — this is the everyday C1-Nano path for residual streams, logits, K/V staging.
  • To the metadata side: writes only (metadata-only RG records addressed by spm_addr). The diagram’s return paths for metadata are the five DSU output queues → CMN Arbiter → CMN (ACP) — a push channel, not a read bus. Metadata reads exist only on the AXI-Lite debug path.

So: C1-Nano software absolutely accesses SPM through CMN — it is the primary software path — but the metadata memory is write-and-be-notified, never demand-read.

Q5

Data path between C1-Nano and the SPM registers

C1-Nano ld/st → DSU (RN-F) → CMN mesh → SBSX CSR node (“NOC access to all other major sub-systems”, ACE5→AXI4 bridge in the CMN wrapper) → SPM AXI4-Lite port (64-bit) → spm_axil_regs (spm_west)

The host reaches the same block through PCIe: in the summarized system address map (HRA components, relative to PF0 BAR2 + 0x2000), SPM owns the 4 KB window 0x234000–0x234FFF, beside FAST Regs (0x233000), SAM (0x235000), SAL (0x236000) and SVL (0x237000). Behind this window sit the control register (enable, sw_reset, interrupt enables, bypass-mode knobs, new_c2c_tx_send_to_flags), write/read/delivery counters, bank-conflict capture registers, interrupt status/enable/force, and the indirect RG-memory access registers from Q3. One exception: the redundant-bank remap is not behind AXI-Lite — it is a separate 2-wire serial chain (1204 bits, programmed once at boot).

Known fields of an RG record

From the SPM uArch spec (names, no bit offsets yet): send_to_sal, send_to_c2c, send_to_cmn, acp_dsu_sel (1-of-5 DSU), spm_addr (entry address for CMN metadata-only writes), c2c_spm_rwrite_addr (same, for C2C), spm_source (2 b, stamped by SPM: 0=SVL, 1=SAL, 2=C2C, 3=CMN); plus the routing identifiers named in the arch spec (model, matmul, user, TP group/ID, C2C Tx flag) and optional Generated Data (top-k values+indices, sum, max).

Two hardware rewrites your software will observe: SPM stamps spm_source on ingest, and on delivery to C2C Tx it forces send_to_cmn=1, send_to_c2c=0, send_to_sal=0 (configurable via new_c2c_tx_send_to_flags; disable via disable_force_c2c_tx_send_to_flag) so the record lands on a remote peer pre-routed to that peer’s ARM cores.

Gap to be aware of: the bit-level layout of the 46 B record is defined in neither document. The arch spec’s three field tables are literal placeholders (“<insert Routing Data here>”), and the ACP notification field list is likewise “<insert fields here>”. Until the Interface Spec / a later SPM draft pins it down, software cannot parse RG records from first principles. Also note a small internal inconsistency: the SPM spec’s overview once lists the seven queues as “CMN, C2C and five DSUs”, while every detailed section says SAL, C2C, five DSUs — the latter is consistent with the target decoder and read-port assignments.

Q7

Word length, address length, endianness

PropertyData memoryMetadata (RG) memory
Word length256 B (2048 b); 32 tiles × 64 b slices46 B (368 b) in 384 b row; 6 tiles × 64 b
Capacity64 K words = 16 MiB (+4 K spare = 17 MiB)64 K entries = 3 MiB (+spare = 3.19 MiB)
Word address16-bit, shared: [3:0] bank · [5:4] segment · [15:6] row — consecutive words stripe across 16 banks
Byte address (CMN view)24-bit (16 MiB); write granule 16/32 Bn/a — addressed per 46 B entry only
Access unitSVL/SAL/C2C: 256 B · CMN: 16 B or 32 Bwhole entry

Endianness: unspecified. Neither the SPM uArch spec nor the arch spec states byte order for SPM data or metadata; the only endianness statement anywhere is the P4_1 “little-endian-nibble” weight packing for WMem, which is unrelated. In practice the system is little-endian end-to-end (AArch64 C1-Nano cores in their default byte order, byte-invariant AXI/CMN interconnect), so bytes written by the host or ARM cores read back in the same order. But because vectors are produced by hardware (SVL element ordering within a 256 B word, BF16 byte order, RG field packing), element-within-word layout is a spec gap worth confirming with the SPM/SVL designers — flagged as an open item below.

What is transparent to software — and what is not

MechanismTransparent?What software sees
4-cycle write / 3-cycle read round-robin; 8 kb write & 6 kb read turn accumulationYesLatency/throughput only.
Bank conflicts on CMN trafficYesHW pauses and retries next turn; chunk selection dodges back-to-back conflicts. Perf effect only. (The hard conflict-free-address rule binds SVL’s write streams, i.e. address planning in command generation — not C1-Nano accesses.)
Segments, tiles, control-chain pipelining, 2 GHz decode stagingYesNothing; flat address space.
Bypass vs store-and-forward metadata modes; virtual queues & creditsYesOrdered ACP arrivals. Debug knobs exist (disable_rg_data_bypass_mode, update_rg_mem_in_bypass_mode).
Full-2 kb internal reads behind 16/32 B AXI readsYesFunctionally invisible; wastes array bandwidth on small scattered reads.
spm_source stamping & C2C-Tx send_to rewriteNoDelivered records carry these HW-modified fields — parse accordingly.
Redundant-bank remap (1204-bit serial chain) & SRAM assist registersNo (boot)Boot software must shift the chain from the good/bad-bank map before any SPM traffic; no read-back. Transparent afterwards.
SPM control.enable, interrupts, countersNoRG delivery is off until enable=1; conflict/FIFO-error interrupts and capture registers are software-visible state.

Open items to confirm with the design team

  • Bit-level layout of the 46 B RG record and the ACP notification payload (placeholders in both specs; the token-walk docs also flag this).
  • Element/byte ordering conventions inside a 256 B vector word as produced by SVL/TWE (endianness is stated nowhere).
  • How the CMN AXI address window distinguishes data writes from metadata-only writes (wr_rg_addr_fifo width “16 or 26” hints at a separate decode; not spelled out).
  • Cacheability/shareability attributes ARM software should use for the SPM data window (SLC interaction is not specified).
  • The queue-list typo (“CMN” vs “SAL” in the overview) — worth a one-line erratum in the next SPM draft.
